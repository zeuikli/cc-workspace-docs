# Spec-Driven Development with Claude Code in Action

## 核心摘要

Alexander Opalic 呈現了一個四階段工作流，用於使用 Claude Code 代理系統管理複雜軟體重構。與其直接深入實作，該方法將 Claude 視為「AI 開發團隊」，人類擔任產品所有者。

該方法論解決 AI 輔助編碼的兩個關鍵問題：**上下文污染**（累積失敗嘗試填滿上下文窗口）和**代理遺忘**（會話間進度喪失）。Opalic 透過利用 Claude Code 的任務持久化系統解決了此問題，該系統將任務儲存為 `.claude/tasks/` 中的 JSON 檔案，使工作能在會話重啟中倖存。

## 四階段工作流

**第一階段：平行研究**產生多個子代理同時調查參考程式碼庫的不同方面。在 Opalic 的 IndexedDB 遷移專案中，五個代理獨立探索 Jazz 框架的架構，報告 CRDT、WebSocket 協議、儲存模式等的發現。

**第二階段：規格建立**將研究整合為綜合技術文件（`indexeddb-migration-spec.md`）。此規格成為「真實來源」，防止分散實作任務的不一致性。

**第三階段：規格精煉**使用 Claude 的 `AskUserQuestion` 工具在實作開始前浮現設計歧義。Opalic 質疑衝突解決策略、跨標籤同步方法和 Vue 特定模式。

**第四階段：實作**透過任務委派。主會話擔任協調者，建立生成新鮮子代理上下文的任務。「在每個任務後進行提交再繼續」確保原子、可審查的變更。

## 關鍵技術洞察

任務系統的創新在於**依賴性對應**。任務參考 `blocks` 和 `blockedBy` 屬性，讓 Claude 識別可平行化的工作。這防止「上下文填滿失敗嘗試」問題，因為每個子代理以清晰記憶啟動。

Opalic 強調**背壓**：預提交鉤子執行測試和 linting 自動立即捕捉錯誤。「系統驗證正確性，你專注於高層決策」，而非手動審查成為瓶頸。

## 可測量結果

SQLite 至 IndexedDB 遷移通常需要 2-3 天手動工作。使用此工作流，Opalic 約 45 分鐘內完成，具有 14 個任務、14 個提交和 15+ 個修改檔案。儘管協調多個子代理，上下文使用仍維持在 71%，證明委派模式的效率。

## 何時應用此模式

Opalic 推薦大型重構、遷移、不清楚需求和學習外部程式碼庫的規格驅動開發。它不適合小型錯誤修復或定義明確的單一檔案變更。

## 實踐建議

1. **從研究開始**：平行代理調查參考實作發現人類可能遺漏的模式
2. **編碼前文件化**：寫好的規格倖存會話重啟並指導子代理一致性
3. **提早詢問利害關係者**：在規格階段澄清歧義，而非實作階段
4. **原子提交**：每個任務產生一個 git 提交，簡化回滾
5. **使用背壓**：自動化驗證在來源捕捉問題，而非累積

## 關鍵引用與數據

- 遷移時間改進：2-3 天 → ~45 分鐘
- 任務數：14 個任務
- 提交數：14 個提交
- 修改檔案數：15+ 檔案
- 上下文使用：71%（儘管協調多個子代理）
- 子代理數：5 個（平行研究階段）

## 核心建議

此工作流鏡像企業軟體工程實踐——平行工作、移交、依賴追蹤和品質閘門。Opalic 將其定位為簡單 AI 提示與完全自主系統（如在無狀態 bash 迴圈中執行的 Ralph）之間的中點。任務系統提供足夠結構以實現一致性，無需完全自主代理的設置開銷。

該方法從根本上重新構想人類-AI 協作：停止將 Claude 視為獨奏編碼者，改為將其架構為協調團隊，人類管理協調和高階決策。
