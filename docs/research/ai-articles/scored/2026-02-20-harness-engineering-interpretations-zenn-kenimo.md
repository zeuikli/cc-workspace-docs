# ハーネスエンジニアリング、全員が違うことを言っている — 5社の解釈を並べてみた

## 「ハーネスエンジニアリング」、定義がバラバラ問題

OpenAIが2026年2月に「Harness engineering」を公開してから、この用語が急速に普及しました。Anthropicは2つのガイドを発表し、LangChainは公式ブログで定義し、martinfowler.comではBirgitta Böckeler氏が解説を執筆し、arXivには関連論文も投稿されています。

しかし、すべての関係者が微妙に異なることを述べています。同じ「ハーネス」という言葉を使いながら、比喩表現はステアリングから馬具、車体へと多岐にわたり、乗り物図鑑のような様相を呈しています。

## まず共通認識：ハーネス ⊇ コンテキスト ⊇ プロンプト

すべての論者が同意する入れ子構造が存在します。SmartScopeの記事によると、「ドキュメントにリンター実行を記載すること」と「Hookで実行を強制すること」の間には、「ほぼ毎回」と「例外なく毎回」という違いがあります。

この点まではコンセンサスがありますが、ここからが議論の分かれ目です。

## OpenAI：「宣言を書け。コードは書くな」

OpenAIの記事は衝撃的でした。5ヶ月間、エンジニアは「1行もコードを書かず」、100万行を超える本番アプリケーションをCodexエージェントのみで構築しました。ビルド時間は手作業の10分の1に短縮されました。

OpenAIの見解では、「人間がステアリングを行い、エージェントが実行する」ため、ハーネスとは「宣言的制約」です。つまり、「あるべき状態」を記述し、実装をエージェントに委ねるというアプローチです。

重点は以下にあります：
- 大規模プロジェクトのスケーリング
- エージェントの並列実行
- サンドボックス環境での安全な実行

## Anthropic：「コンテキストを管理しろ。モデルは不安になる」

Anthropicは2つの公式ガイドを発表していますが、出発点がOpenAIと異なります。OpenAIが「大規模プロジェクト全体の自動化」から始めたのに対し、Anthropicは「長時間動作するエージェントの安定性」から出発しています。

Anthropicは「context anxiety（コンテキスト不安）」という独自概念を導入しています。AIも過度な情報量で不安定になるということです。

重点は以下にあります：
- コンテキスト管理による不安回避
- セッション引き継ぎを含むライフサイクル管理
- シングルエージェント設計への簡素化

## LangChain：「Agent = Model + Harness。数字で証明する」

LangChainの公式ブログは最もシンプルな定義を提供しています。「エージェント」とは「モデル＋ハーネス」であり、モデルは知能をもたらし、ハーネスがその知能を実用的にするとしています。

重要な点として、ハーネス設計改善のみで、ベンチマーク精度が「52.8%から66.5%」に向上したと定量的に示されました。

重点は以下にあります：
- モデル非依存の設計原則
- 定量的実証
- オーケストレーションとオブザーバビリティツール

## Birgitta Böckeler（martinfowler.com）：「コードベース自体がハーネスだ」

Böckeler氏は全く異なる角度を提示しています。強い型付き言語で書かれたコードベースは「自然に型チェックがセンサーになる」と指摘しています。

つまり、AGENTS.mdを記述する前に、コードベース自体がハーネスの一部として機能しているということです。TypeScriptのstrict modeやRustのBorrow Checkerは、意図せずしてエージェントの品質管理として機能します。

重点は以下にあります：
- コードベース自体が本質的に持つ制約
- 型安全性とテストの価値の再認識
- ハーネスは後付けではなく内在するもの

## arXiv論文：「ハーネスを形式仕様化せよ」

学術的アプローチではさらに異なる視点が提示されています。ハーネスパターンロジックを「読み取り可能かつ実行可能なオブジェクト」として外部化すべきと主張しています。

重要な知見として、モデルの性能向上にかかわらず、「役割、契約、検証ゲート」といったハーネスレベルの制御が自然言語で指定されることは継続的に価値を持つと述べられています。

## 横断比較：何が同じで、何が違うか

### 全員が同意していること

1. モデルの外側が重要
2. 制約は「お願い」ではなく「強制」すべき
3. フィードバックループが必須
4. プロンプトエンジニアリングは廃止にはならない

### 意見が分かれていること

**マルチエージェント vs シングルエージェント：**
- OpenAI：並列スケーリング
- Anthropic：モデルの進化によりシングルで十分

**ハーネスの粒度：**
- OpenAI：プロジェクト全体で1つのハーネス
- Böckeler：型チェック1つもハーネス

**「置き換え」か「積み重ね」か：**
- 置き換え派：エージェント能力がプロンプト設計の想定を超えた
- 積み重ね派：本質的な違いはなく、LLM能力向上の反映

## 結局、実務者はどうすればいいのか

解釈の相違は興味深いですが、実行段階は単純です。

**ステップ1：** AGENTS.mdまたはCLAUDE.mdを作成（500文字程度で十分）

**ステップ2：** 品質ゲートを自動化（リンター、型チェック、テストをhooksで強制）

**ステップ3：** フィードバックループを実行（エージェントの誤り → AGENTS.mdに制約追加 → 次回以降の改善）

これら3ステップで、あなたはハーネスエンジニアリングを既に実践しています。各社の解釈の違いは、ステップ3を3ヶ月実行した後に検討すれば十分です。

## 繁體中文全文摘要

**原文語言**：日文（分析五個主要組織對「Harness Engineering」的不同詮釋）

### 核心命題：同一個詞，五個組織說的是不同的東西

OpenAI 在 2026 年 2 月發布「Harness engineering」後，這個術語迅速普及。但 OpenAI、Anthropic、LangChain、Birgitta Böckeler（martinfowler.com）、arXiv 論文——每個人說的都略有不同。

### 共識基礎：Harness ⊇ Context ⊇ Prompt

所有論者都同意這個嵌套結構。「文件中記載執行 linter」vs「透過 Hook 強制執行」的差異——「幾乎每次」vs「每次無例外」——這個點有完整共識。分歧從這裡之後才開始。

### 五大詮釋對比

#### OpenAI：「寫宣告，不要寫程式碼」
- 5 個月、0 行手寫程式碼、100 萬行以上的生產應用，由 Codex agents 獨立構建
- Harness = **宣告式約束**（描述「應有狀態」，把實作委託給 Agent）
- 重點：大規模專案擴展、Agent 並列執行、沙箱環境安全執行
- 人類負責 steering，Agent 負責執行

#### Anthropic：「管理 Context，模型會不安」
- 出發點：「長時間運作 Agent 的穩定性」（vs OpenAI 的「大規模自動化」）
- 獨特概念：**Context Anxiety（Context 不安）**——資訊量過大時 AI 也會不穩定
- 重點：透過 context 管理避免不安、含 session 交接的生命週期管理、簡化為單一 Agent 設計

#### LangChain：「Agent = Model + Harness，用數字證明」
- 最簡定義：Agent = 模型（提供智能）+ Harness（讓智能實用化）
- 量化驗證：**純靠 harness 設計改善，benchmark 精準度從 52.8% 提升至 66.5%**
- 重點：模型無關設計原則、量化實證、Orchestration 和 Observability 工具

#### Birgitta Böckeler（martinfowler.com）：「Codebase 本身就是 Harness」
- 完全不同的角度：以強型別語言撰寫的 codebase 天然讓型別檢查成為感測器
- TypeScript strict mode、Rust Borrow Checker——這些不是後加的 harness，而是**內在的品質管理**
- 重點：Codebase 本身固有的約束、型別安全和測試的重新認識、Harness 是內在的不是後加的

#### arXiv 論文：「形式化 Harness 規格」
- 應把 harness 模式邏輯外部化為「可讀且可執行的物件」
- 關鍵洞見：無論模型效能如何提升，「角色、合約、驗證閘門」等 harness 層級控制用自然語言指定仍持續有價值

### 橫向比較

**全員同意的點**：
1. 模型外側很重要
2. 約束應「強制」而非「請求」
3. 反饋循環是必要的
4. Prompt Engineering 不會廢止

**意見分歧的點**：

| 面向 | OpenAI | Anthropic |
|------|--------|----------|
| 代理架構 | 多 Agent 並列擴展 | 單一 Agent（模型進化使其足夠）|
| Harness 粒度 | 整個專案一個 harness | 一個型別檢查也算 harness |
| 演化方向 | 置換（新能力超越舊設計假設）| 積層（LLM 能力提升的反映）|

### 實務三步驟

詮釋的分歧雖有趣，但執行階段其實很簡單：

1. **建立** AGENTS.md 或 CLAUDE.md（約 500 字元就足夠）
2. **自動化**品質閘門（linter、型別檢查、測試透過 Hooks 強制）
3. **執行**反饋循環（Agent 犯錯 → AGENTS.md 加約束 → 下次以後改善）

這三步就是在實踐 Harness Engineering。各組織詮釋的差異，執行第三步三個月後再研究就夠了。
