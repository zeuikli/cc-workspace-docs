# Claude Codeでハーネスエンジニアリングを実践する — 5層の設計パターン

## はじめに

Claude Codeを効果的に運用するには、プロンプト作成だけでは不十分です。AIの動作を環境側で設計・管理する「ハーネスエンジニアリング」というアプローチがあります。CLAUDE.md、Skills、Agents、Hooks、Settingsといった複数の仕組みを組み合わせることで、AIエージェントを意図した方向に動かせます。

著者は個人用のナレッジ管理リポジトリでこの設計パターンを実践しており、その知見を紹介する記事です。

## ハーネスエンジニアリングとは

### 概念の成り立ち

ハーネスエンジニアリングはHashiCorp共同創業者のMitchell Hashimotoが2026年2月のブログで提唱した概念です。「エージェントがミスをするたびに、同じ失敗を繰り返さないよう環境側に恒久的な修正を施す」という思想に基づいています。

この概念はOpenAIやAnthropicなど主要企業により発展させられ、Martin Fowlerはフィードフォワード制御（Guide）とフィードバック制御（Sensor）の枠組みで体系化しました。

### LLMの本質とハーネスの必要性

LLMは本質的に「ステートレスな関数」です。推論中にモデルの重みは変わらず、内部に記憶機能がありません。同じ入力でも出力が異なるのはサンプリング時の確率的ゆらぎに起因します。

重要な点として、「LLMは推論中に何も覚えない」という特性があります。会話が続いているように見えるのは、アプリケーション層が毎回全会話履歴をプロンプトに含めているからです。つまり、毎回がLLMにとって初対面の状態です。

プロジェクトのルール、過去の失敗事例、ディレクトリ構成など、すべてを毎回外部から注入する必要があります。この注入の仕組み全体を設計することがハーネスエンジニアリングです。

### 3つのエンジニアリングの位置づけ

| 概念 | 対象 | 焦点 |
|------|------|------|
| プロンプトエンジニアリング | モデルへの入力テキスト | 「何を聞くか」の最適化 |
| コンテキストエンジニアリング | モデルに渡す情報全体 | 「何を見せるか」の最適化 |
| ハーネスエンジニアリング | モデルの外側の実行環境全体 | 「どう動かすか」の最適化 |

コンテキストエンジニアリングはハーネスの一部です。

## Claude Codeにおけるハーネスの構成要素

| 要素 | 役割 | ファイル |
|------|------|----------|
| CLAUDE.md | プロジェクト全体のルール・コンテキスト | `CLAUDE.md` |
| Rules | 特定領域のドメインルール | `.claude/rules/*.md` |
| Skills | 再利用可能なワークフロー定義 | `.claude/skills/*/SKILL.md` |
| Agents | 専門特化したサブエージェント | `.claude/agents/*.md` |
| Settings | ツール権限・セキュリティ制御 | `.claude/settings.json` |

これらは層として積み重なってClaude Codeの動作を制御します。

## 5層の実装レイヤー

### Layer 1: CLAUDE.md — プロジェクトの「憲法」

CLAUDE.mdはClaude Codeが最初に読み込む指示書です。プロジェクトの目的、ディレクトリ構成、技術スタック、守るべきルールを記述します。

著者のリポジトリでは以下を記載しています：

- リポジトリの目的（情報管理と意思決定支援の2軸）
- ディレクトリ構成と配置指針
- やらないこと（意思決定自動化、ファイル整理自動化など）
- 技術ルール（Python実行は`uv run`で、パッケージバージョン固定）
- セキュリティルール（APIキーのハードコーディング禁止など）

重要なのは「やらないこと」を明記することです。これは推論の抑制ではなく、行動スコープの定義です。境界線があるだけで、不要な変更が劇的に減ります。

#### 個人設定のimport方式

チーム開発ではCLAUDE.mdがgit管理されるため、個人の好みを直接書き込めません。CLAUDE.mdの末尾に以下を追加できます：

```
@my-preferences.md
```

`my-preferences.md`を`.gitignore`に追加すれば、各メンバーが個人設定を管理できます。参照先ファイルが存在しない場合は無視されるため、まだファイルを作っていないメンバーに影響はありません。

### Layer 2: Rules — ドメイン特化のルールブック

`.claude/rules/`にMarkdownファイルを置くと、Claude Codeが自動的に読み込みます。CLAUDE.mdの肥大化を防ぎ、ドメインごとにルールを分離できます。

読み込みタイミングはフロントマターの`globs`指定で変わります：

| 条件 | 読み込みタイミング | コンテキスト消費 |
|------|------------------|------------------|
| `globs`なし | セッション開始時に常時ロード | 常時 |
| `globs`あり | マッチするファイルを読んだ時点で遅延ロード | 条件付き |

例：
```yaml
---
description: API層のコーディングルール
globs: "src/api/**/*.ts"
---
```

著者は以下の2つのルールファイルを運用しています：

- **github_projects.md** — タスク管理ルール（カンバン構成、粒度、利用方法）
- **output_format.md** — 出力ルール（命名規則、ファイル構成、データ形式）

### Layer 3: Skills — 再利用可能なワークフロー

Skillsは`/コマンド名`で呼び出せるワークフロー定義です。SKILL.mdにMarkdownでプロンプトを書くだけで複雑な手順を実行できます。

著者のリポジトリでは13個のスキルを運用しており、代表例は：

**情報収集系：**
- `/tech-daily2` — Pythonで事前収集し、LLMは翻訳・分類のみ
- `/new-books` — 技術書新刊を出版社ベースでチェック
- `/career-research` — 日本市場の企業リサーチ

**深掘り・分析系：**
- `/company-research` — 企業調査とEDINET財務分析
- `/pdf-translate` — PDF処理と日英翻訳

**ナレッジ管理系：**
- `/rag-index` — MarkdownをDuckDBにembedding格納
- `/rag-ask` — RAGインデックスをベクトル検索

重要な設計原則は、「決定的な処理をスクリプトに、判断と生成をLLMに」という責務分離です。初期版の`tech-daily`ではすべてをLLMに任せていましたが、改善版の`tech-daily2`ではPythonスクリプトがRSS取得・パースを行い、LLMは翻訳と分類だけを担当します。結果、トークン消費が1/6、処理時間が1/3に削減されました。

### Layer 4: Agents — 専門特化のサブエージェント

Agentsは`@名前`で呼び出す、特定役割に特化したサブエージェントです。メインのClaude Codeとは別のコンテキストで動作するため、大量情報を扱うタスクでもメインコンテキストを汚しません。

著者のリポジトリでは、リポジトリ内の蓄積情報を横断検索・要約する専門エージェントを運用しています。曖昧な問い（「〜について何かあったっけ？」）に対応できます。

利点はコンテキスト分離です。数百ファイルを検索する場合、メインコンテキストで実行するとウィンドウが埋まってしまいますが、エージェント委譲なら要約だけが返ってきます。

### Layer 5: Settings — 権限とセキュリティの制御

`settings.json`でツールの使用権限を制御します。これはハーネスの「安全装置」です。

例：
```json
{
  "permissions": {
    "allow": [
      "Bash(gh issue *)",
      "Bash(gh project *)",
      "Bash(jq *)",
      "Bash(uv run*python3 .claude/skills/*)",
      "WebSearch",
      "WebFetch",
      "Write"
    ],
    "deny": [
      "Bash(gh repo delete*)",
      "Bash(gh repo archive*)",
      "Bash(gh pr close*)"
    ]
  }
}
```

設計ポイント：

- **allowは業務に必要な範囲で広く** — 頻繁に使うコマンドは許可して作業のリズムを保つ
- **denyは破壊的操作を明示的に禁止** — リポジトリ削除やアーカイブなどを防ぐ
- **settings.jsonとsettings.local.jsonの使い分け** — 汎用ツールはgit共有、URL固定のcurlコマンドはローカル専用

## 設計で学んだこと

### 「やらないこと」を定義する重要性

CLAUDE.mdに「やらないこと」を明記するのは最も費用対効果の高い設計判断です。LLMは放っておくとファイル整理やドキュメント追加など「改善」と称して余計なことをします。明確な境界線があるだけで、これが劇的に減ります。

### LLMとスクリプトの責務分離

最初はすべてをLLMに任せていましたが、決定的な処理（API呼び出し、データパース、ファイルI/O）をPythonに移すことで、トークン消費と安定性が大幅に改善しました。LLMの仕事は「判断」と「生成」に限定するのが理想です。

### CLAUDE.mdの肥大化を防ぐ

肥大化するとコンテキストを圧迫し、指示の優先度も曖昧になります。最初からドメインルールはRulesに分離する方針が効果的です。CLAUDE.mdは「憲法」であり「百科事典」にすべきではありません。

### パーミッション設計は実運用で磨く

最初からすべてを正しく設定するのは不可能です。「実行して確認が出る → 許可パターンを追加する」サイクルで、徐々に快適な設定に収束させることが現実的です。

### Hooksはまだ使っていない

このリポジトリではHooksを未導入です。CLAUDE.mdやRulesは「お願い」（advisory）、Hooksは「強制」（deterministic）という性質が異なります。

Hooksでできることは幅広いですが、ブロック用途では`settings.json`の`deny`で十分なケースが多いです。Hooksが必要なのは、「変更対象ファイルの中身を見てバリデーションする」など、パターンマッチでは表現できない動的な判断が求められる場面です。

## まとめ

ハーネスエンジニアリングはClaude Codeを「賢いチャットボット」から「自分専用の開発環境」に変えるアプローチです。

CLAUDE.mdでプロジェクトの憲法を定め、Rulesでドメインルールを分離し、Skillsで繰り返すワークフローを定義し、Agentsで専門的タスクを委譲し、Settingsで安全装置を整備することで、Claude Codeは会話するたびに文脈を理解し、ルールに沿って動く「自分のチーム」になります。

ハーネスが整っていれば、扱える領域はどこまでも広がります。

## 繁體中文全文摘要

**原文語言**：日文（日本工程師分享 Claude Code Harness Engineering 5 層設計實踐）

### 核心命題：LLM 是無狀態函數，Harness 負責注入記憶

LLM 本質上是「無狀態函數」——推論中模型權重不變，內部沒有記憶功能。對話看起來持續，是因為應用層每次把完整對話歷史注入提示。**每次對 LLM 來說都是初次見面**。

因此，專案規則、過去失敗案例、目錄結構——一切都必須每次從外部注入。設計這個注入機制的整體就是 Harness Engineering。

### 三層工程的定位

| 概念 | 對象 | 焦點 |
|------|------|------|
| Prompt Engineering | 對模型的輸入文字 | 「問什麼」的優化 |
| Context Engineering | 傳遞給模型的資訊整體 | 「展示什麼」的優化 |
| **Harness Engineering** | **模型外側的執行環境整體** | **「如何運作」的優化** |

Context Engineering 是 Harness 的一部分。

### 5 層設計架構

| 層次 | 角色 | 檔案 |
|------|------|------|
| **CLAUDE.md** | 專案整體規則與 context（「憲法」） | `CLAUDE.md` |
| **Rules** | 特定領域的 Domain 規則 | `.claude/rules/*.md` |
| **Skills** | 可重用的工作流定義 | `.claude/skills/*/SKILL.md` |
| **Agents** | 專門化的子代理 | `.claude/agents/*.md` |
| **Settings** | 工具權限與安全控制 | `.claude/settings.json` |

### Layer 1：CLAUDE.md 是「憲法」

最重要的設計判斷：**明確記載「不做的事情」**。

LLM 若放任不管，會以「改善」之名進行檔案整理、加文件等多餘動作。只要有明確的邊界線，這種情況會大幅減少。「不做的事」定義的是行動範圍，不是推理抑制。

個人設定分離方式：CLAUDE.md 末尾加 `@my-preferences.md`，並將該檔放入 `.gitignore`。

### Layer 2：Rules 的條件式載入

`.claude/rules/*.md` 的讀取時機由 YAML frontmatter 的 `globs` 控制：

| 條件 | 載入時機 | Token 消耗 |
|------|---------|-----------|
| 無 `globs` | Session 開始時常駐載入 | 常時 |
| 有 `globs` | 符合的檔案被讀取時延遲載入 | 條件付き |

例：API 層的 coding rule 只在處理 `src/api/**/*.ts` 時載入。

### Layer 3：Skills 的責務分離原則

關鍵設計原則：**決定性處理交給 script，判斷與生成交給 LLM**。

初版 `/tech-daily`（所有工作都給 LLM）→ 改善版 `/tech-daily2`（Python script 負責 RSS 取得/解析，LLM 只負責翻譯和分類）：
- Token 消耗減少 **1/6**
- 處理時間減少 **1/3**

### Layer 4：Agents 的 Context 隔離

子代理在與主 Claude Code 不同的 context 中運作，處理大量資訊時不會污染主 context。數百個檔案的搜尋任務，透過子代理委派只有摘要回傳。

### Layer 5：Settings 的安全設計

- **allow**：業務需要的操作範圍廣泛許可（保持作業節奏）
- **deny**：破壞性操作明示禁止（repo 刪除、archive、PR close 等）
- **分工**：`settings.json`（通用工具，git 共享）vs `settings.local.json`（URL 固定的 curl 命令，本機專用）

### 實踐心得

1. **CLAUDE.md 肥大化預防**：Domain 規則從一開始就移到 Rules；CLAUDE.md 是「憲法」，不是「百科全書」
2. **Permission 設計靠實際運作磨礪**：「執行 → 出現確認 → 加許可模式」循環，逐步收斂到舒適設定
3. **Hooks 是強制（deterministic），Rules 是建議（advisory）**：只需要 Pattern Match 無法表達的動態判斷時才用 Hooks
