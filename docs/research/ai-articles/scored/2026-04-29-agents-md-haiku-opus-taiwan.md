## 核心貢獻

### 1. 在地化詮釋（最有價值之處）

**原文**「好文件能把 Haiku 升級到 Opus」 →  
**本文詮釋**：拆解為「模型選擇迷思」的破解

```
常見誤解：最強模型 = 最好結果
正確認識：中階模型 + 清晰流程 > 強模型 + 模糊需求
```

作者自身背景（多年 AI 開發經驗）讓這個論點有說服力，不只是理論。

### 2. 分層教學

按讀者認知階段：
1. 開場：與讀者共識（「大家常問什麼模型」）
2. 引入：Augment Code 研究為什麼重要
3. 核心 7 點：逐點白話拆解
4. 組織延伸：跳脫技術層，談企業能力

### 3. 組織洞見的提升

原文偏技術工程師視角；本文加上管理視角：

> AGENTS.md 是一種新的組織能力：把資深同仁腦裡的默契，轉成 AI 可理解、可執行的操作系統。

**含義轉變**：從「寫給 Agent 的文件」→ 「組織工作系統的新形式」

## 七點白話詮釋（對照原文）

| 原文模式 | 本文詮釋 | 新增洞見 |
|---------|---------|---------|
| Progressive Disclosure | 不要一次塞太多 | 類比「新進員工 onboarding」 |
| Procedural Workflows | 把任務拆成清楚步驟 | Agent 需要「一條清楚的路」 |
| Decision Tables | 遇到選擇題先寫判斷標準 | 把選擇「提前處理」而非「臨場猜測」 |
| Production Examples | 給 3-10 行真實程式碼 | 「學寫作看 2 篇好範例勝過 10 篇雜亂」 |
| Domain Rules | 規則要具體，不漂亮話 | 「資料品質」vs 「payment_logs 必須同步」 |
| Pair + Solutions | 不只禁止，要給替代方案 | Agent 不是不認真，是被提供替代資訊 |
| Modular Docs | 模組化參考 | 避免「全部丟進去期待自然變強」的陷阱 |

## 壞文件的三大危害（新增細節）

1. **過度探索** — 文件太大拉走 Agent 注意力
2. **語氣模糊** — 「優先」vs「必須」Agent 理解不穩定
3. **過時文件最危險** — Agent 會很認真實作過期的架構決策

**特別提醒**：「壞文件不中性，它主動傷害結果」

## 引申思考（本文獨有）

### 「寫到 Agent 看得懂 = 真的想清楚」

> 當你能把一件事寫到 AI Agent 看得懂、做得到、驗證得出來，通常也代表你真的把這件事想清楚了。

這是一個強有力的清晰度測試：**能給 Agent 的指令 = 真正理解的標誌**

### 模型升級 vs 環境升級

結尾的反思很關鍵：
```
不要急著只升級模型，也要升級 Agent 工作的環境。
```

對應 workspace 的實踐：CLAUDE.md 設計逐年迭代，比每年換新模型的效果更持久。
