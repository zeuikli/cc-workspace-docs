# The Reality of Vibe Coding: AI Agents and the Security Debt Crisis

## 核心摘要

Reya Vir 的文章深入探討了「氛圍編程」（Vibe Coding）——即利用 AI 代理快速生成程式碼的開發方式——所帶來的安全隱患。作者透過實例和研究數據，揭示了 AI 編程助手在追求速度時如何犧牲安全性。

### 核心安全問題

文章以 Moltbook 社交網絡為引例，說明即便是看似引人注目的 AI 項目，也可能因基礎安全漏洞而崩潰。該平台因配置不當的 Supabase 資料庫而泄露了 150 萬個 API 密鑰和 3.5 萬個用戶電郵地址。根據哥倫比亞大學的研究，AI 編碼代理失敗的主要源於三個關鍵問題。

**速度優先於安全性** — AI 語言模型被優化用於獲得用戶接受，導致它們移除驗證檢查、放鬆資料庫策略或禁用身份驗證流程，只為消除運行時錯誤。這種面向結果的思維方式完全忽視了安全含義。

**缺乏對副作用的認知** — AI 常因無法完整理解大型複雜代碼庫的上下文，而在修復一個檔案的錯誤時，卻在其他引用該檔案的地方造成破壞性變化或安全漏洞。這種跨檔案依賴性理解的缺失是 AI 代理的根本弱點。

**模式匹配而非判斷力** — 語言模型本質上只是預測下一個最可能出現的 token，並不真正理解程式碼的語義或影響。它們不知道為什麼存在安全檢查，只知道移除它們能讓程式碼運行。

### 三個實際案例

作者列舉了常見的安全問題：

1. **API 密鑰泄露** — AI 將 OpenAI 密鑰直接寫在 React 前端檔案中，使其在瀏覽器開發工具中可見。

2. **資料庫公開訪問** — 面對權限拒絕錯誤時，AI 建議使用 `USING (true)` 策略，使整個資料庫暴露於網絡。

3. **跨站腳本攻擊（XSS）漏洞** — AI 直接使用 `dangerouslySetInnerHTML` 呈現原始 HTML 內容，而不建議使用如 dompurify 的清理庫。

## 解決方案

文章提出三項改進建議。

**更好的提示工程** — 不應籠統地要求 AI「確保安全」，而應採用規範驅動開發，預先定義安全策略。文章建議基於 OWASP Top 10 制定政策，並使用思維鏈提示，要求 AI 在編寫程式碼前推理安全含義。

**更徹底的審查** — 儘管氛圍編程承諾無需檢查程式碼，但目前還不可行。開發者應檢查程式碼差異、單元測試和程式碼品質，特別是安全相關邏輯。

**自動化防護機制** — 在 CI/CD 管道中添加安全掃描器，自動阻止包含硬編碼密鑰或危險模式的提交。GitGuardian 和 TruffleHog 等工具可自動掃描暴露的密鑰。
