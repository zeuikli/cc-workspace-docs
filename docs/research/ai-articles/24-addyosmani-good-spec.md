# How to Write a Good Spec for AI Agents

## 核心摘要

Addy Osmani 的文章提供了一套完整框架，用於撰寫能有效指導 AI 編碼代理產出高品質結果的規格。核心主題針對一個常見挫折：將大量規格文檔傳遞給 AI 模型會產生差劣結果，因為上下文窗口限制和注意力約束。

### 五項核心原則

**高層次願景優先**
不應在前期過度工程化需求，而應從簡潔的目標陳述開始，讓 AI 擴展規格。這充分利用 LLM 的改善優勢，同時保持方向控制。Osmani 建議從「計劃模式」開始——一個只讀階段，代理分析現有程式碼並草擬詳細計劃，然後再編寫任何內容。這防止了未等規格穩定就倉促進入代碼生成的陷阱。

**像專業文檔一樣結構化**
應將規格視為正式產品需求文檔，包含六個關鍵部分：可執行命令、測試程序、專案結構、程式碼風格範例、Git 工作流程、明確邊界。針對 2,500+ 代理配置檔案的 GitHub 分析顯示，有效的規格一致地涵蓋了這些領域。與其模糊描述，應包括具體細節：「React 18 搭配 TypeScript、Vite 和 Tailwind CSS」優於「React 專案」。規格成為「人類和 AI 都不斷參考的共享真實來源」。

**模組化上下文優於單體提示**
研究表明過多指令反而降低模型表現——「指令詛咒」。應將規格分成聚焦部分，每項任務只提供相關上下文。Osmani 描述了使用摘要延伸的目錄、階層結構，甚至針對不同領域的專門子代理。這防止上下文超載，讓 AI 專注於單一問題而非同時應付多項需求。

**建置自查和約束**
實施三層邊界系統：✅ 總是做（自動）、⚠️ 先問（需人類批准）、🚫 絕不做（硬停止）。此細緻方法優於單純的禁止清單。納入驗證步驟，鼓勵代理根據規格自我審計。「絕不提交密鑰」在 GitHub 研究中成為最有幫助的約束。透過編碼已知陷阱、特定庫警告和邊界案例，利用領域專業知識直接進規格。

**持續迭代**
應將規格撰寫視為循環而非一次性工作。早期且頻繁測試，蒐集反饋，精煉規格，並利用支援自動化檢查的工具。在版本控制中追蹤規格本身，像對待程式碼一樣對待。考慮運用檢索增強生成或上下文管理工具，聰慧地浮現相關規格部分。此循環——編碼、測試、反饋、精煉——大幅改進結果。

## 關鍵實務例子

Osmani 展示了四階段工作流：指定（用戶體驗焦點）、計劃（技術架構）、任務（具體工作單位）、實作（帶有審查門檻的執行）。每個階段在進行前都驗證，防止 Osmani 稱之為「紙牌屋程式碼」的現象——脆弱的輸出在審視下崩潰。

規格結構範本展示了組織：
- 清晰的目標陳述
- 明確的技術堆棧細節
- 帶標誌的完整命令
- 展示偏好風格的具體程式碼範例
- 特定資料夾組織

## 關鍵警告

Osmani 指出常見失敗：模糊規格（「建造酷東西」）、跳過人類審查儘管通過測試、未總結地上下文傾倒、混淆快速原型製作與生產工程。「致命三角」的速度、非確定性和成本鼓勵走捷徑——嚴格規格和審查流程必須抵禦這些壓力。

## 結論

有效的 AI 代理規格需要調整傳統軟體工程原則以適應 LLM 特性。透過從清晰開始、專業結構化、維持模組化焦點、建置驗證機制、擁抱迭代，開發人員將 AI 協助從混亂轉變為受控。規格本身成為「可執行製品」，驅動實作而非靜態文檔蒙塵。
