## 核心貢獻

首篇明確區分「Scaffolding（組裝期）」與「Harness（執行期）」的論文，以 OPENDEV（Rust 終端機代理）為案例。

## 概念定義

| 概念 | 時機 | 職責 |
|------|------|------|
| **Scaffolding** | 執行前（組裝期） | 系統提示、工具定義、subagent 註冊 |
| **Harness** | 執行期（執行迴圈） | 工具分派、context 管理、安全執行、session 持久化 |

## OPENDEV 架構創新

1. **工作負載專門化的模型路由**: 不同任務類型路由到最適合的模型
2. **懶惰工具發現（Lazy Tool Discovery）**: 減少不必要的工具載入，保護 context budget
3. **適應性 Context 壓縮**: 動態調整壓縮策略，非固定閾值
4. **雙代理架構**: Planning agent + Execution agent 分離
5. **自動化記憶系統**: 事件驅動的跨 session 記憶持久化

## 核心洞見

- Scaffolding 是靜態配置，Harness 是動態執行時
- 大多數 harness 改進來自**結構層面**（工具、中介軟體、記憶），而非提示詞措辭
- Lazy Tool Discovery 可顯著降低 context 占用，同時提升工具選擇精確度

## 對設計的啟示

- 工具列表不應在 session 開始時全部載入
- Harness 的複雜度應根據任務類型動態調整
- Planning 與 Execution 分離可以提升整體品質

## 與本 Workspace 的關聯

- `defer_loading: true` 的工具設計 ≈ Lazy Tool Discovery
- plan-mode-expert + implementer 的分工 ≈ 雙代理架構
- `.claude/settings.json` 的 hooks ≈ 事件驅動記憶系統
