# 優化七個 Token 消耗陷阱，Claude Code 可多用三倍

## TL;DR

絕大多數 Claude Code 用戶的額度耗盡不是因為模型變弱，而是系統 overhead 膨脹導致。精簡七大陷阱（CLAUDE.md、會話歷史、插件 Hook、Skill/MCP 膨脹、快取機制、全局 Extended Thinking、未及時停止回答）可將有效產出倍增至 2-3 倍。

---

## 核心論點

### 1. Overhead 才是真兇
- 產出 token 僅占總消耗 30%，70% 被預扣費用吃掉
- 預扣項目：CLAUDE.md 反覆加載、會話歷史指數重讀、插件 Hook 群體注入、缺失或失效快取、Skill/MCP schema 常駐

### 2. 七大量化陷阱（按吃 token 重量級排序）

| 陷阱 | 消耗 | 修復方式 |
|------|------|---------|
| **1. CLAUDE.md 膨脹** | 單會話 100 萬+ token | 砍至 <1500 token；下沉項目級、抽 Skill、刪未用規則 |
| **2. 會話歷史重讀** | 第 N 條成本是第 1 條的 N 倍 | 硬上限 20 條消息；錯誤時編輯而非追加；用 /compact |
| **3. 插件 Hook 偷塞** | UserPromptSubmit × 4 = 6000+ token | 查 .claude/settings.json；禁無法說清理由的 Hook |
| **4. 快取 5 分鐘失效** | 600+ 次失效事件 | 升級 1 小時快取（2 倍寫入成本，10+ 恢復可回本） |
| **5. 備而不用的 Skill/MCP** | Skill 13,500 token；MCP 7,000+ token | 7 日審計日誌；禁零調用者；MCP 保留 3 常用 |
| **6. Extended Thinking 全局開** | 小任務 3000+ thinking token | 預設關閉，按消息按需開（Alt+T） |
| **7. 跑偏回答未停止** | 350 行浪費輸出 | Cmd+. / Ctrl+. 立刻停止；5 秒內決策 |

### 3. 操作簡單可量化
每個陷阱都有 30 秒上手的修復方法（禁用、刪除、改設定）；對應 bash / jq 一鍵審計工具現成可跑。

### 4. 複合收益驅動 2-3 倍產能
- 單次節省累加：一筆改動受益於所有後續請求
- 高頻放大：日均請求次數越多，複利越明顯
- 範例計算：CLAUDE.md 節省三成 baseline + 會話重讀節省四成 + Hook 節省五成 ≈ 每週 200 請求額外省百萬 token

### 5. 心智模型升級
會話不是白紙而是發票——每個 CLAUDE.md、Hook、Skill、MCP 都是前置扣費。用戶需主動審計而非被動接受預設。

## 關鍵數據 & 實驗設計

### 作者數據蒐集方法
- HTTP 代理架設於 Claude Code + Anthropic API 之間，攔截完整請求
- 跟踪數百萬級 input token 記錄（時間戳、prompt、模型、token 數）
- 將所有 token 按用途分類（產出 vs overhead）

### 量化實測結果

**前後對比（作者親測）：**

| 項目 | 修前 | 修後 | 改善 |
|------|------|------|------|
| CLAUDE.md 大小 | 4,800 token | 900 token | 81% ↓ |
| 會話平均長度 | 60+ 條消息 | 15 條消息 | 75% ↓ |
| UserPromptSubmit Hook | 4 個 | 1 個 | 每 prompt -5,800 token |
| SessionStart Hook | 9 個 | 2 個 | 每次會話 -1,200 token |
| Skill 啟用數 | 11 | 4 | 削減 63% |
| MCP 連接數 | 12 | 3 | 削減 75% |
| 快取失效影響 | 600+ 次 | 大幅下降 | 升級 1h 快取回本 |
| **產出占比** | **30%** | **65%** | **+217% ↑** |
| **Overhead 占比** | **70%** | **35%** | **50% ↓** |

### 官方確認與支持證據

**Anthropic 官方（3 月底）：**
- 承認：用戶撞額度限制速度比預期快「許多」
- 快取 bug：某些會話實際成本膨脹 10-20 倍（GitHub issue #40524）
- 高峰配額下調：影響約 7% 用戶

**社群回報：**
- Max 5 訂閱：19 分鐘耗盡一天額度（預期 5 小時）
- Pro $200 用戶：30 天內僅可用 12 天（周一滿，周六才重置）

### 審計工具（現成腳本）

```bash
# 掃出全部 7 陷阱
wc -w ~/.claude/CLAUDE.md .claude/CLAUDE.md  # 陷阱 1
cat ~/.claude/settings.json | jq '.hooks.UserPromptSubmit'  # 陷阱 3
grep "skill_invoked" ~/.claude/logs/*.log | sort | uniq -c  # 陷阱 5
# 其餘陷阱對應 /mcp、快取設定、Extended Thinking 全局開關查詢
```

### 對標常見「網傳偏方」效果評估

| 建議 | 實際幫助 | 為何效果有限 |
|------|---------|-------------|
| 簡單任務用 Haiku | 3% | 上下文膨脹才是主因；便宜模型跑膨脹上下文比貴模型跑精簡上下文還貴 |
| 每個任務間 /clear | 反作用 | 丟掉真正需要的上下文；反而迫使重新解釋 |
| 禁所有 Skill | 短期有效，淨虧 | 不得不手動重寫 200+ token 指令；長期成本更高 |
| 避開高峰時段 | 7% | 僅影響受高峰配額調整的用戶；前述七個槓桿更重要 |

## 批判性觀察

### 論點支撐充分性（評級：高）
- **實驗設計嚴謹**：HTTP 代理、百萬級日誌、完整分類——結論可量化驗證
- **透明性**：公開了修復前後的各項數據、計算方法、邊界假設
- **缺陷**：未經同儕審查；未公開原始日誌供獨立驗證

### 過度簡化風險（評級：中等）
- **應用邊界**：主要針對「重度使用者」（每週多次撞限額）；輕度使用者收益有限
- **使用模式差異**：未討論不同工作流（代理開發 vs 文本分析）對 overhead 組成的差異
- **快取 ROI 假設**：「10 次快取恢復回本」假設用戶行為模式相對固定；實際可能波動

### 實踐易用性（評級：高）
- 修復皆為操作級別（禁用、刪除、改設定），無需架構重構
- 審計工具現成可跑，結果即時可見
- 每項修復都標註「30 秒上手」的可操作性

### 適用邊界清晰（評級：高）
- 專門針對 Claude Code 官方客戶端；API-only 用戶不適用
- 大量篇幅聚焦快取、Hook、Skill 機制——均為客戶端特有約束
- 不涉及模型層或 Anthropic 後端優化

## 工作區可行動性（優先序）

1. **審計 CLAUDE.md + 規則檔** → 比對 1,500 token 上限；超額則下沉項目級、抽 Skill、刪未用規則
2. **禁不必要的 Hook** → UserPromptSubmit、SessionStart；審視 .claude/settings.json；保留 1-2 常用
3. **統計 Skill 調用** → 7 日審計日誌；刪除零調用者；保留 3-4 真正在用的
4. **檢查 MCP 連接** → /mcp 列表；禁不常用項；保留 3 常用
5. **會話管理習慣改進** → 20 條消息上限、優先編輯而非追加、/compact 而非 /clear
6. **快取策略升級** → 評估成本（2 倍寫入、0.1 倍讀取）；若日常工作 10+ 快取恢復則回本
7. **Extended Thinking 全局關閉** → 按消息按需開（Alt+T）；試一次不滿意時才開
8. **停止生成習慣養成** → Cmd+. / Ctrl+. 5 秒內決策；別等「後面說不定就對了」

## 整合決策

### 5 維度評分

| 維度 | 權重 | 評分 | 說明 |
|------|------|------|------|
| A. 工作區可行動性 | 30% | 9/10 | 配置級改動；審計工具現成；立即可見效果 |
| B. 創新性 | 20% | 6/10 | 「Overhead 為主因」中等創新；修復為標準最佳實踐組合 |
| C. 證據品質 | 20% | 8/10 | 真實數據（代理、百萬級日誌）；方法論透明；可重現 |
| D. 技術深度 | 15% | 7/10 | 快取、Hook、Draft.js、複利；未涉及內部實現 |
| E. 泛化性 | 15% | 6/10 | 重度用戶專款；輕度收益有限；API 用戶不適用 |

**加權總分**：9×0.3 + 6×0.2 + 8×0.2 + 7×0.15 + 6×0.15 = **7.75 / 10**

### 分類建議

**決策**：**SKILL 級別**（而非 Rule 或單純文檔）

**理由**：
1. 工作流為多步驟自主流程（審計 → 分類 → 刪除/禁用 → 驗證）
2. 指導對象專款（Claude Code 重度用戶）且場景反覆（周期性審計）
3. 適合自動檢查腳本 + 互動式導引（vs 靜態規則）
4. 決策邏輯複雜（「何時禁用、何時保留」需上下文）

**建議 Skill 能力**：
- 審計模式：一鍵掃描 7 陷阱，返回簡報
- 修復向導：逐項引導修復（「刪除這些 Hook？」「升級快取嗎？」）
- 量化驗證：修復前後成本對比預估
