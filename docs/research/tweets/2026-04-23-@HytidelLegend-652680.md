## 全文

以 Claude Code 为例，对其他 agent 产品也适用。24 個類別，80 種省 token 方法。

### 核心方法（精選重點）

#### 1. 退訂

退訂所有你還不知道怎麼用的 AI 產品——從源頭切斷浪費。

#### 2. 配好環境再開干

- 提前配好環境，不要用 Claude Code session 配置環境
- **會話中保持配置不變**：確定 CLAUDE.md、工具列表後再開始，防止 KV Cache 失效

#### 3. CLAUDE.md 優化

- 合理安排全局 vs 項目級 CLAUDE.md（項目特定的工程規範放項目級，不要塞入全局）
- CLAUDE.md 需包含項目文件樹+每個文件說明，避免 Claude 自行遍歷
- 刪去與系統提示詞重疊的部分

#### 4. Skills、MCP 優化

- 全局和項目都不要放太多 skills（節約讀 metadata token；避免 skill 描述重疊導致詢問確認）
- 全局只放通用 skills（如 skill-creator、文件讀取）
- 項目級按需放置：當前步驟用到的 skills，其他移走
- 保持 skill description 簡潔
- 利用中間產物持久化調試（調試後去掉非必要中間產物輸出）
- 在 settings.json 設置 MCP 最大輸出 token 數
- 關閉暫時用不到的 MCP servers

#### 5. .claudeignore

屏蔽 node_modules/ 等不需要讀取的目錄。

#### 6. 提示詞優化

- **優先使用英文**（比其他語言省 token）
- **消融處理**：每句話思考「去掉後執行會改變嗎？」——不會就刪
- **反向約束**：有時「不要做 XX」比「要做 A、B、C」更短
- **Few-shot 示例**替代冗長描述
- **XML 區分**文字說明和示例

#### 7. 模型分級使用

- Opus：複雜任務
- Sonnet：大部分情況
- Haiku：極其簡單任務、定時任務心跳
- **Advisor Strategy**：Sonnet/Haiku 作為執行者，Opus 作為顧問（遇到無法解決的問題才請教）
- 調節 Effort：簡單任務不要設過高 effort

#### 8. 會話管理

- **及時 /compact 或清理**：context > 70% 時模型可能「變蠢」，建議壓縮（可指定保留內容）
- 監控上下文：向 Claude Code 發送指令查看佔用
- 接下來的操作與上文完全無關 → 新開會話
- **任務一次性發送**（同時有多個問題 → 一次性發送，context 只增長一次）
- **同一 session 保持模型不變**（切換模型 = 無緩存重發整個對話歷史，成本驟增）
- **無關提問用 /btw**（讀取上下文但不追加到對話歷史）
- 禁用 recap 功能

#### 9. 規劃

在 Claude Code 執行前先進行規劃，避免走錯路後再重做。

#### 10. 開啟 Subagent

複雜任務分解為多個子任務，分別由 subagent 執行，各自維護獨立 context。

#### 11-24. 其他優化

- 檔案格式：選擇 token 效率高的格式
- 確定性任務：採用腳本執行而非 LLM
- **緩存**：保持 CLAUDE.md、工具列表在會話中不變（prompt caching）
- 漸進式披露（Progressive Disclosure）
- 精簡模型輸出
- API 調用優化
- 本地搜索引擎
- 白嫖每日免費額度（用其他 agent 做環境配置等輔助任務）

## 研究摘要

**TL;DR**：24 個類別 80 種 Claude Code Token 節省方法，最有技術含量的是 KV Cache 維護策略（不換模型、不改工具列表）、Advisor Strategy 架構、/btw 指令防止 context 污染、以及消融處理的 prompt 設計原則。

### 核心論點

- KV Cache 前綴穩定性是最高槓桿的省 token 方法：同 session 保持模型、工具列表、CLAUDE.md 不變
- Advisor Strategy 是已被本 workspace 採用的模式（Sonnet 執行 + Opus 顧問），本文提供了更清晰的定義
- Context > 70% 模型「變蠢」→ 此閾值與 cc-workspace 的 context-management.md（70% 通知）完全吻合
- 消融處理（ablation）的 prompt 設計哲學：每句話都要問「去掉後結果會改變嗎？」

### 關鍵數據 / 工具

- Context 70% 閾值：與 cc-workspace 規則獨立驗證一致
- /btw 指令：讀取上下文但不追加到對話歷史（防 context 污染）
- settings.json 中可設置 MCP 最大輸出 token 數（具體配置鍵名未說明）
- 模型分級：Opus/Sonnet/Haiku 三級

### 批判性觀察

- 文章有商業推廣成分（推薦第三方 API 中轉站「￥1 換 $1」），中立性存疑
- 「80 種」的計數未逐一列出，實際上很多是同類別的變體
- 「同 session 保持模型不變」的建議與 cc-workspace 的 context-management.md 中 Prompt Caching 架構規則一致，可作為交叉驗證
