#### 一、記憶系統進階

**1. Hermes 記憶系統整體架構**

三層：**內置記憶（始終激活）** + **外部記憶提供商（可選，叠加）** + **運行時上下文**

內置記憶的兩個文件（預設存在 `~/.hermes/memories/`）：

- **MEMORY.md**：Agent 工作筆記（環境事實、項目約定、學到的技巧）
  - 系統閾值：2,200 字符（硬限制）
  - 建議日常維持在 1,800 字符（約 1,200 漢字），預留 20% 給自動追加
- **USER.md**：用戶畫像（偏好、溝通風格、工作習慣）
  - 系統閾值：1,375 字符（硬限制）
  - 建議日常維持在 1,100 字符（約 700 漢字）

關鍵設計：**冷凍快照（Frozen Snapshot）** — 文件在每次 session 開始注入，**不**實時更新。原因：
- 節省 Token：實時更新破壞 KV Cache 前綴穩定性，成本和速度都劣化
- 防止記憶污染：只有 Agent 判斷「重要」的內容才被策展寫入

**2. Agent-curated Memory 機制：為什麼它「不記得」？**

觸發記憶寫入的場景：
- 明確表達偏好（「我喜歡/不喜歡 xxx」）
- 環境事實（「這台機器裝了 xxx」）
- 糾正 Agent 錯誤（「不要用 sudo，我在 docker 組里」）
- 完成重要任務里程碑
- 明確要求記住某件事

寫入節奏：結合 `nudge_interval` 控制，**不**是每句話實時觸發。

解決方案：明確下達記憶指令 — 「記住我的偏好：所有代碼統一使用 Python 3.11，不要用 3.12 或 3.13」

**3. 核心記憶文件正確使用方法**

## 研究摘要

**TL;DR**：Hermes Agent v0.9.0 進階指南，最有技術深度的部分是記憶系統架構（Frozen Snapshot 設計、字符硬限制、nudge_interval 機制），以及多 Agent 協作的 Sub-Agent spawn 和 Profiles 隔離。

### 核心論點

- Hermes 記憶「不記得」的本質：冷凍快照設計 + Agent-curated（非全量記錄），為 KV Cache 效率和記憶品質的平衡
- 字符硬限制是重要的工程細節：MEMORY.md 2,200 字符、USER.md 1,375 字符，超過截斷
- Frozen Snapshot 的設計邏輯（KV Cache 前綴穩定性）是通用的 LLM Agent 設計原理，不限於 Hermes
- 多 Agent 協作的 Profiles 功能（同機器多獨立 Agent，各自隔離記憶）是值得借鑑的架構模式

### 關鍵數據 / 工具

- Hermes Agent v0.9.0（本文基準版本）
- MEMORY.md 字符限制：2,200（硬限制），建議 1,800（留 20%）
- USER.md 字符限制：1,375（硬限制），建議 1,100
- nudge_interval：記憶策展的節奏控制參數
- 外部 Memory Providers（可選叠加，非替代）

### 批判性觀察

- Hermes Agent 不是 Anthropic 官方產品（是第三方工具），但其記憶架構設計原則對理解 Claude 的記憶機制有類比價值
- Frozen Snapshot + KV Cache 設計的解釋是本文最有技術含量的部分，與 cc-workspace 的 context-management.md 相互印證
- 多 Agent Profiles 隔離概念可對應到 cc-workspace 的 subagent 策略
