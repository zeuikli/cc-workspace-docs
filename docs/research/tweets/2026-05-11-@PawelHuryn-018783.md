## TL;DR

知識層（knowledge layer）的複利效應來自三原則：可攜性優於供應商鎖定、確定性優於自主性、語境優於控制。

---

## 核心論點

1. **知識層不是一個肥大的指令檔**，而是「一個常駐 orchestrator + 按需載入的文件集 + 工具 registry + 跨 session 結構化資料（假設、規則、決策）」的組合。
2. **可攜性（Portability）**：CLAUDE.md 用 `@AGENTS.md` 一行引用，讓 Claude Code、Codex、Cowork 讀同一份內容；「五分鐘換 harness 測試」是鎖定風險的判斷基準。
3. **確定性（Determinism）**：判斷邏輯（when/what/why）留在 markdown，機械性步驟（API 包裝、JSON 解析、寫檔）封裝成工具。規則：任何「不說 it depends 就能描述的步驟」應成為 code。
4. **語境優於控制（Context > Control）**：告訴 agent *為什麼*，而非只給命令；agent 擁有 codebase 維護權（提議規則更新、晉升工具、記錄假設），人只留在 review 路徑。
5. **三原則互相強化**：高 agency prompt（Context）→ agent 提議新工具包裝（Determinism）→ 標記 harness 漂移（Portability）。

## 關鍵架構 / 模板

### 知識層目錄結構
```
my-project/
├── CLAUDE.md             # orchestrator（常駐，指向其餘）
├── AGENTS.md             # one-liner: @CLAUDE.md（供 Codex）
├── knowledge/
│   ├── INDEX.md          # 路由系統（按需載入地圖）
│   ├── pricing/
│   │   ├── knowledge.md  # 事實與模式
│   │   ├── hypotheses.md # 待更多數據
│   │   └── rules.md      # 已確認，預設套用
│   ├── onboarding/
│   ├── social-media/
│   └── competitors/
└── tools/                # 自定工具 registry
    ├── xai.py
    ├── x_api.py
    └── telegram_send.py
```

### 知識層 CLAUDE.md 塊（可直接使用）
```markdown
## Knowledge Layer

Before starting a new task, review existing rules and hypotheses for this domain.
Apply rules by default. Check if any hypothesis can be tested with today's work.

At the end of each task, extract insights. Store them in domain folders, e.g.:
  /knowledge/pricing/         (or /onboarding/, /competitors/)
    knowledge.md  (facts and patterns)
    hypotheses.md (need more data)
    rules.md      (confirmed — apply by default)

Maintain a /knowledge/INDEX.md that routes to each domain folder.
Create the structure if it doesn't exist yet.
When a hypothesis gets confirmed 5+ times, promote it to a rule.
When a rule gets contradicted by new data, demote it back to hypothesis.
```

### 工具晉升觸發規則（Preferred Tools 塊）

```markdown
For example, when the same fetch/parse logic comes up more than once, suggest wrapping

<!-- List project-specific tools here. For each, link to its skill or script file. -->
```

### Collaboration 高 agency 塊
```markdown
## Collaboration

- When you notice a recurring friction, a missing workflow, or a better way to
  organize how we work — say so. Don't wait to be asked.
- Update the knowledge layer proactively, immediately. When the next move on
  `knowledge/` is obvious, just edit the file.
- Default to acting. Ask when a question is cheap and useful — before big token
  spends, irreversible moves, or non-obvious choices between approaches.
```

## 批判性觀察

1. **個人經驗為主，缺乏可量化指標**：文章以「三個月重建三次」的個人歷程為主要證據，雖然有真實工具範例（screenshot via CDP），但沒有 session 數量、錯誤率改善等可重現數據。
2. **假設晉升為規則的門檻（5 次）過於武斷**：「確認 5 次升為規則」在不同任務密度下意義差異很大；缺乏關於如何界定「確認」的說明。
3. **Skills vs. custom markdown 比較停在表面**：「Skills win when you're publishing, custom markdown wins when you're building」這個二元結論過於簡化；兩者混用的實際操作細節缺失。

## 5 維度評分

| 維度 | 分數 | 理由 |
|------|------|------|
| A. Workspace 可行動性 | 8/10 | 提供可直接貼入 CLAUDE.md 的三個模板 block；hypothesis→rule 晉升機制與此 workspace 結構高度相容 |
| B. 創新性 | 7/10 | Judgment-in-markdown / mechanics-in-tools 分離框架清晰；假設狀態機（active/graduated/rejected）在現有 skills 未見完整框架 |
| C. 證據品質 | 5/10 | 個人經驗敘述為主，有具體工具範例但無實驗數據或 benchmark |
| D. 技術深度 | 7/10 | 包含真實目錄結構、可複製的 markdown 模板、CDP screenshot 工具範例，細節可直接參照 |
| E. 泛化性 | 8/10 | 跨 harness（Claude Code/Codex/Cowork）設計，適用所有 agentic workspace，非特化情境 |

**加權總分**: 8×0.3 + 7×0.2 + 5×0.2 + 7×0.15 + 8×0.15 = **7.05/10**

## 整合決策

**Rule** — 核心是跨任務通用原則（Portability/Determinism/Context 三框架），建議更新 `.claude/rules/core.md` 補充 hypothesis 狀態機與工具晉升觸發規則。
