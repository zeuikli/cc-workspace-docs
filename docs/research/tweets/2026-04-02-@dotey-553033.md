## 全文

团队内的 skills 管理和维护的一点经验分享（以 codex cli 用的 .agents/skills 目录为例）：

1. Git 管理是一定要的。
版本控制太重要了，而且 Skills 都是 Markdown 和脚本文件，天然适合 Git。

2. 用好 Symlink。
不要把 Skills 整个拷贝到 .agents/skills，而是通过 Symlink 直接链接到原始 Skills 的 Repo。

好处有两个：一是版本控制更干净；二是使用中遇到问题，Agent 定位后可以直接在 Repo 里改，改完就能 Review 提 PR。

我日常维护 baoyu-skills 就是这么干的，用的时候发现问题，让 Agent 在当前会话改，改的就是 Repo 本身，流程非常顺。

主要的坑是 Windows 下好像不支持 Symlink，另外首次配置稍麻烦（可以让 Agent 帮你操作）。

3. Skills 的编辑优先让 Agent 来。
改完走 Git 提 PR，这就是最好的协作和发布流程。

4. 验证确实不太好做。
脚本部分可以写单元测试，Skill 的 Markdown 部分只能靠平时积累的测试集，大部分还得人工。
但配合 Git 的版本管理，快速迭代反而更现实：不怕改出问题，出了问题根据 commit history 快速定位，或者直接回滚。

5. 最后提醒一下：大部分 Skills 应该跟着项目走（放项目目录下的 .agents/skills），不要放全局（~/.agents/skills）。
即使是渐进式加载，meta 信息累积起来也会占不小的上下文空间。

## 研究摘要

**TL;DR**：分享 Skills 團隊管理的 5 個實踐：Git 版本控制必要、用 Symlink 連接到原始 Repo（不是複製）、優先讓 Agent 編輯 Skills 後 PR、測試主要靠人工加測試集、大部分 Skills 應放在專案層級而非全局。

### 核心論點
1. **Git + Symlink 是核心工作流**：不把 Skills 複製到 `.agents/skills`，而是用 symlink 指向原始 Repo — 版本控制更乾淨，Agent 定位問題後直接在 Repo 改並 PR
2. **Agent 協助維護 Skills**：遇到問題讓 Agent 在當前 session 直接改 Repo，改完就可 review 提 PR — 人機協作形成閉環
3. **驗證仍然是痛點**：腳本部分可以單元測試，Markdown 部分只能靠人工 + 測試集累積；快速迭代 + Git 回滾比追求完美更現實
4. **專案層級優先**：大部分 Skills 應放 `.agents/skills`（專案目錄），不放 `~/.agents/skills`（全局）— 即使漸進式加載，meta 累積也佔 context
5. **Windows Symlink 坑**：Windows 不支援 Symlink（或需要特殊權限），是重要的跨平台限制

### 關鍵數據 / 工具
- 實踐者：dotey (宝玉)，個人維護 baoyu-skills Repo
- 適用工具：Codex CLI 的 `.agents/skills` 目錄（與 Claude Code `.claude/skills/` 同概念）
- 630 likes · 778 bookmarks · 68,287 views
- Windows 問題：Symlink 在 Windows 需要管理員權限或開發者模式

### 批判性觀察
- Git + Symlink 方案的複雜度比直接複製高，對新手門檻較高
- 「讓 Agent 改 Repo 然後 PR」的流程假設 Agent 了解 Git workflow，需要 CLAUDE.md 明確規定 PR 流程
- 「大部分 Skills 應跟著專案走」的建議與 skill-authoring.md 的 progressive disclosure 設計一致，可相互驗證
- cc-workspace 的 `.claude/skills/` 已符合建議的專案層級模式
