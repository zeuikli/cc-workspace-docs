## 核心主張

LLM 工程從 Prompt Engineering → Context Engineering → Harness Engineering，不是線性替代而是**螺旋積層**：每一圈包含前一圈作為基礎設施。每次演化都由兩個力量同步驅動：模型更強了、人敢多放手一點。

**Harness Engineering 的定義**：
> 為 agent 構建運行環境的工程實踐。第一步是**放手**：給 agent 工具和反馈通道，讓它自主獲取信息、自主判斷。第二步是**上保險**：給 agent 框定能力邊界，確保自主運行不失控。

## 全文內容

虽然 LLM 应用的历史始于 2022 年 ChatGPT 问世，但工程实践已经从 Prompt Engineering，经历 Context Engineering，发展到如今的 Harness Engineering。想要彻底理解 Harness Engineering，就必须回到前面两者，理解为什么旧的做法不够用了，新的做法解决了什么，又卡在了哪里。

**TL;DR**

- Prompt Engineering：任务是静态的，人手工优化单次输入。局限：解决不了多步任务。
- Context Engineering：任务是动态展开的，程序按预设规则编排步与步之间的信息流转。局限：人硬编码的规则限制了模型的自主能动性。
- Harness Engineering：模型足够强了，放手让它自主获取反馈、自主判断，同时框定能力边界作为保险。

---

## Prompt Engineering：对着🔮许愿的时代

2022 年底 ChatGPT 上线，所有人第一次面对同一个问题：怎么让这个东西给出有用的回答？

这个时期出了不少今天看来相当魔幻的事情。有人发现在 prompt 末尾加一句"这对我的职业生涯非常重要"，模型的输出质量就会提升。有人发现说"请一步步思考"比直接问答案靠谱得多。

这些尝试逐渐沉淀出一套实践：Few-shot、Chain-of-Thought、Role assignment、Output format。

**Prompt Engineering 的局限**：本质上是单次任务的优化。然后模型开始支持工具调用，任务从"问一个问题"变成了"跑一个流程"。瓶颈从"怎么问"移到了"给它看什么"。

---

## Context Engineering：当任务不再是一句话能说清的

真实世界的任务很少是静态的。修一个 bug 的完整过程：搜日志 → 定位模块 → 读代码 → 修函数 → LSP 报 type error → 改调用方 → 跑测试 → 发现边界情况。每一步的输出决定了下一步该做什么、该看什么。

**Context Engineering 要回答的问题**：
- 上一步执行完了，它的结果怎么流入下一步的 context？
- 环境产生了反馈（LSP 报错、测试失败），怎么注入？
- 对话历史越来越长，哪些压缩、哪些保留？
- 外部检索回来的信息，在哪一步注入、注入多少？

Cursor 是典型的 Context Engineering 产品：根据正在编辑的文件动态检索相关代码注入，不是把所有文件都塞进 context。

**2025-06-25 Karpathy 的定义**：
> "Context engineering is the delicate art and science of filling the context window with just the right information for the next step."

注意他用的动词：filling。是**外部在填**，不是模型自己在找。

**Context Engineering 的天花板**：
- Microsoft + Salesforce 研究：多轮对话推进，模型性能平均下降 39%，即使是最强的 o3 也从 98.1% 掉到 64.1%（context rot）
- 更根本的问题：当模型能力超过人类硬编码的规则时，人来做信息编排反而成了瓶颈

---

## Harness Engineering：放手，然后上保险

### 放手：把信息获取能力交给 agent

**Khan 的 Prompting Inversion 证据**：Sculpting 精细 prompt 技法在 gpt-4o 上准确率 97%（超过 CoT 的 93%），但在 gpt-5 上反转了（CoT 96%，Sculpting 降到 94%）。模型越强，人类精心设计的技巧反而成了约束。

**对比 Context Engineering vs Harness Engineering**：
- Lovable（Context Engineering）：程序自动捕获 LSP 报错，机械地塞进下一轮 context，agent 被动接收
- Stripe Minions（Harness Engineering）：agent 拥有 lint/测试/代码搜索工具，自己决定什么时候用、用哪个

### 上保险：框定能力边界

Sandbox 隔离、CI 轮数限制、文件权限限制、架构约束（违反模块边界直接 fail）。能力边界不是对模型的不信任，而是对非确定性系统的工程纪律。

### 什么不算 Harness

状态持久化、compaction、session 续接、跨 context window 的记忆管理 → 这些是 agent **runtime** 该做的事，不是 harness。Harness 关心的是 agent 跑的时候能不能自主获取反馈、会不会被约束住。

## 三大案例

### Stripe Minions

每周合并超过 1000 个 PR，全部 agent 写的，人只做 review。

**放手**：Agent 可以跑 lint、跑测试、通过 Sourcegraph 搜索代码、通过 MCP 查阅内部文档和 ticket。设计原则："shift feedback left"，5 秒内完成本地 lint 检查。

**上保险**：跑在隔离 devbox，不能访问生产环境和外网；最多跑两轮 CI；MCP 连接 400+ 内部工具，但每个 agent 只能访问经过筛选的子集。

**注意**：Stripe 的按子目录自动加载 rule files → 这是 **context engineering**（程序预设），不是 harness。

### Anthropic Claude Agent SDK 实验

让 agent 自主构建完整 web 应用（claude.ai 克隆）。

**三种失败模式 → 对应保险**：

| 失败模式 | 保险机制 |
|---------|---------|
| 一口气做太多 | 每个 session 只做一个 feature（硬约束） |
| 过早宣布完成 | feature_list.json 全标 failing，agent 只能改 passes 字段（用 JSON 不用 MD，结构更难被篡改） |
| 假装测试通过 | 必须用 Puppeteer 做端到端验证 |

**设计原则**：能力边界从失败模式倒推出来，不是凭空画线。

### OpenAI Codex 团队

3-7 人，5 个月，100 万行代码，1500 个 PR。

**Ryan Lopopolo 总结**：
> "Agents aren't hard; the Harness is hard."

**设计亮点**：定期"垃圾回收"——持续扫描整个代码库，发现架构 drift 自动创建修复任务给 agent。

**上保险**：分层架构由 structural tests 强制执行（不是靠 prompt 说"请不要违反架构"）。

**最终结论**：
> "Our most difficult challenges now center on designing environments, feedback loops, and control systems."

## 定量验证

- **Epsilla 实验**：同一个模型、同一个 prompt、同一份数据，只换 harness 配置 → 任务成功率从 **42% → 78%**
- **LangChain Terminal Bench 2.0**：同一个 Opus 4.6 模型，不同 harness 下 solve rate 差异**超过 60%**

---

## 回头看：一个螺旋的三圈

每一次演化都由三件事同步驱动：
1. 模型更强了一点，能做更长更复杂的任务
2. 人敢多放手一点，从全程在场到只看结果
3. 新的可靠性问题暴露，催生新的工程实践来兜底

**螺旋积层**：前一圈不消失，变成基础设施。Harness 里还在跑 context engineering pipeline，pipeline 里还在写精心设计的 system prompt。

**下一圈已露出**：
- **Eval**：agent 跑完了，结果到底好不好？LLM-as-judge 有 bias，"评估评估的系统"是递归问题。Cisco/Splunk 收购 Galileo 信号明显
- **Governance**：多 agent 互相认证、审计、权限管理。NIST 启动 AI Agent Standards Initiative，计划 2026-Q4 发布 Agent Interoperability Profile 初版
