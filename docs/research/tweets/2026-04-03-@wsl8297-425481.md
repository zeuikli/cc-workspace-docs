你有没有想过一个问题：
同样是 Claude 模型，为什么在 Claude Code 里能写出完整项目，直接调 API 却只会给你讲道理？
同样的大脑，一个是能干活的工程师，一个是只会说话的顾问。
这背后的秘密，藏在一个 95% 开发者都忽略的概念里——Harness。

看不懂这一点的人，注定要在 AI 时代掉队。
## 先问你一个扎心的问题
你花了大价钱订阅 AI 工具，兴冲冲地打开对话框，写了一段精心设计的需求，结果呢？
比如你说：“帮我把这个 React 组件的所有 class 组件改成函数组件，顺便加上 TypeScript 类型”
AI 回你：“你可以这样做：1. 把 class 改成 function；2. 用 useState 替换 this.state；3. 添加 Props 接口定义……”
然后呢？然后就没了。
它不会帮你改，不会给你代码，只会像个技术顾问一样告诉你“应该怎么做”。你得自己一个个文件去改，一行行代码去写。
但同样的任务，在 Claude Code 里：
1. 你说“把 UserProfile.jsx 改成 TypeScript 函数组件”
1. 它先 Read 读取文件，理解组件结构
1. 自动 Edit 重写代码：class → function，state → hooks，加上完整类型定义
1. Bash 运行 npm run type-check 验证类型
1. 3 秒后问你：“改好了，要不要我顺便把相关的测试文件也更新一下？”
同一个模型，天壤之别。
为什么？因为裸 API 只是一个会说话的大脑，而 Claude Code 有一套完整的手脚系统——这就是 Harness.
## Harness 到底是什么？为什么它比模型更重要？
忘掉那些复杂的定义。我用最直白的话告诉你：
Harness = 给 AI 大脑装上手脚的系统。
[圖 1]
就像马具让马从野生动物变成人类的得力助手，Harness 让 AI 模型从“只会说话”变成“真能干活”。
它包裹在模型外面，提供五个核心组件：
[圖 2]
1. 工具（Tools）：AI 的“手”
没有工具，模型只能“说”。有了工具，它能“做”：
- Read：读取任何文件内容
- Edit：精确修改代码的任何一行
- Grep：在整个代码库中搜索
- Bash：执行任何 Shell 命令（这是个图灵完备的逃逸舱）
注意最后一个——Bash 工具。通过它，Claude 理论上能做操作系统能做的一切：安装依赖、运行测试、调用 API、操作数据库。
这意味着什么？Claude Code 的能力上限 = 你操作系统的能力上限。
2. 上下文管理（Context Management）: AI 的“记忆”
Claude 的上下文窗口是 200K tokens。听起来很大？
一个真实编码任务，读 20 个文件、搜索 50 次、执行 30 条命令，对话历史会瞬间膨胀到几十万 tokens。
没有管理，要么爆掉窗口，要么模型开始“失忆”。
Claude Code 的解决方案极其精巧：
- 当对话历史接近 92% 时，触发自动压缩
- 摘要早期消息，但保留最近的完整对话
- 关键操作：每次压缩后，CLAUDE.md、系统提示词、工具定义都会重新注入
这就是为什么你在 CLAUDE.md 里写的规则那么“持久”——不是模型记住了，而是 Harness 每次都重新塞给它。
3. 权限控制（Permissions）：AI 的“安全阀”
想象你给 AI 完全的文件系统访问权限。如果它能随意做任何事，会发生什么？
- 删除整个 node_modules 文件夹？
- 修改生产环境的配置文件？
- 执行 rm -rf / 命令？
这就是为什么需要权限控制。
Claude Code 有三种权限策略：
- Allow（允许）：可以自动执行，无需确认例如：读取代码文件、运行测试
- Deny（拒绝）：直接禁止，绝不允许例如：删除系统文件、修改 .git 目录
- Ask（询问）：需要你确认后才能执行例如：安装新依赖、修改配置文件、发布代码
每次 Claude 想执行操作时，Harness 都会先检查权限。你可以在 CLAUDE.md 里提前设置规则：
```markdown
## Permissions

- Allow: read any file, run tests
- Deny: delete files, modify .env
- Ask: install packages, git push
```
这不是限制 AI 的能力，而是让它在安全的轨道上全速奔跑。
4. 记忆（Memory）：跨会话的持久化
上下文管理解决的是“这次对话”的记忆。但如果你明天重新打开 Claude Code，它还记得你的项目规则吗？
这就是 Memory（记忆）的作用。
它是一个跨会话的持久化存储，让 Claude 记住：
- 项目架构约定：“我们用 Zustand 做状态管理，不用 Redux”
- 代码风格偏好：“组件文件用 PascalCase，工具函数用 camelCase”
- 历史决策：“上次讨论决定用 React Query 处理服务端状态”
- 常用命令：“测试用 npm run test: watch，不用 jest"
这些信息存储在项目根目录的 .claude/memory.json 文件里。每次新对话开始，Harness 都会把这些记忆注入给模型。
Memory 让 Claude 从“临时工”变成“老员工”。
你不需要每次都重新解释一遍“我们项目用什么技术栈”、“我们的命名规范是什么”。它都记得。
[圖 3]
这张架构图清晰展示了 Claude Code 的完整系统：用户通过输入 Prompt 或审批权限与系统交互，Harness 层包含工具系统（Read/Write/Edit/Bash/Grep/Gob/Agent）、权限控制（allow/deny/ask）、Hooks（事件钩子）和 Sessions（会话持久化），核心的 Agentic Loop 协调所有组件，底层则是 Claude 模型负责纯文本生成和工具调用决策。
5. 钩子（Hooks）：可编程的事件触发
Hooks 是最聪明的部分——它让 Claude 在特定事件发生时，自动执行某些操作。
想象你在 CLAUDE.md 里设置了这些规则：
- 如果修改了代码 → 就自动运行 lint 检查
- 如果创建了新组件 → 就自动生成对应的测试文件
- 如果发现了 TypeScript 错误 → 就立即尝试修复
- 如果完成了功能 → 就自动运行完整测试套件
这就像给 Claude 装上了“条件反射”：
传统方式：
- 你：“修改 Button 组件”
- Claude：“已修改”
- 你：“运行 lint”
- Claude：“已运行，发现 3 个问题”
- 你：“修复这些问题”
- Claude：“已修复”
- 你：“运行测试”
- Claude：“测试通过”
有 Hooks 的方式：
- 你：“修改 Button 组件”
- Claude：“已修改代码，自动运行了 lint 并修复了 3 个问题，测试全部通过”
这就像给 AI 装上了“自动驾驶”。
你只需要说目标，它会自动完成所有相关的后续步骤。
## Agentic Loop：让五大组件协同运转的“引擎”
有了这五大组件，还需要一个“引擎”让它们协同工作——这就是 Agentic Loop（智能循环）。
它不是第六个组件，而是让前五个组件协同运转的机制。
[圖 4]
这张流程图展示了 Agentic Loop 的完整工作流程：
1. 接收输入：用户 prompt + 系统 prompt + 工具定义 + 对话历史
1. 模型推理：Claude 分析上下文，生成回复（可能包含工具调用请求）
1. 执行工具：Harness 执行工具并收集结果（包含权限检查）
1. 结果回注：工具结果追加到对话历史，回到步骤②继续推理
1. 返回结果：如果不需要工具调用，直接返回最终结果
传统 API 调用：你问 → 模型答 → 结束。
Agentic Loop：你问 → 模型推理 → 调用工具 → 看结果 → 继续推理 → 再调用工具 → …… → 任务完成。
一个复杂任务可能跑几十轮循环。
比如你让它“重构这个函数”：
1. 先 Read 文件，理解代码结构
1. Grep 搜索这个函数在哪里被调用
1. 分析依赖关系
1. 检查 Permissions：我可以修改这个文件吗？
1. 查询 Memory：这个项目的代码风格规范是什么？
1. Edit 修改函数实现
1. Bash 运行测试
1. 看测试结果，决定是否需要调整
1. 如果失败，再次 Edit 修复
1. 触发 Hooks：自动运行 lint 和类型检查
1. 重新测试，直到通过
这不是一次性生成，这是真正的“思考-行动-反馈”循环。
裸 API 做不到这一点。它只会给你一个“重构建议”，然后等你手动执行。
## 2026 年：Harness 之战已经打响
行业已经意识到一个残酷的事实：
同一个模型在不同 Harness 中的表现差距，远大于不同模型在同一个 Harness 中的差距。
换句话说：Harness 比模型更重要。
几个数据点足以证明：
1. OpenCode 的逆袭
OpenCode（前身 SST）是最成功的第三方 Harness，它用 Client-Server 架构解决了 Claude Code 的“单表面”局限——TUI、桌面 App、IDE 插件、Slack 机器人共享同一个后端。
截至 2026 年 4 月初，OpenCode 拿到了 136K GitHub stars，超过了 Claude Code 本身的 107K stars.
一个第三方 Harness，用户量超过了官方产品。这说明什么？
用户要的不是模型本身，而是更好的 Harness。
2. Anthropic 的“封杀令”
2026 年 1 月，Anthropic 采取了一个争议性举措：封堵了第三方工具通过消费者 OAuth Token 调用 Claude API 的通道。
这直接影响了 OpenCode、Cursor、Windsurf 等工具的用户——他们之前可以用 Claude Pro 订阅的 Token 在这些第三方工具里使用 Claude，封堵后必须购买独立的 API Key。
这件事引发了开发者社区的强烈反弹。
但也揭示了 Harness 生态的核心张力：
- 模型提供商希望控制 Harness 层，因为 Harness 决定了 API 调用量和用户体验
- 第三方 Harness 希望模型层是可替换的商品，因为这样它们才能建立独立的价值
这场战争的本质，是对 AI 应用栈控制权的争夺。
3. 模型正在商品化
Claude、GPT、Gemini、DeepSeek 的能力差距正在缩小。
模型正在变成白菜价的算力商品。
但 Harness 的差异化空间巨大：
- 工具设计的精巧程度
- 上下文管理的智能程度
- 权限控制的安全性
- 执行效率的优化
每一个维度都能产生 10 倍的体验差距。

## 研究摘要

**TL;DR**：Claude Code 核心競爭力解析：Harness 架構（CLAUDE.md + Hooks + Skills + Memory）使 95% 用戶未充分利用；3 個 code block 示範 harness 配置

### 核心論點

- **Harness = Claude Code 真正的競爭力**：非模型本身，而是 CLAUDE.md + Hooks + Skills + Memory 組合讓 Claude 能持續學習和執行複雜任務
- **95% 用戶只用了表面功能**：大多數人把 Claude Code 當作聊天機器人，沒有設計 harness
- **三層 Harness 設計**：①CLAUDE.md（規則/約束）②Hooks（自動化/守衛）③Skills（可複用工作流）
- **為什麼 harness 重要**：讓 Claude 在長任務中保持一致性；減少重複指示；實現可驗證的輸出

### 關鍵數據 / 工具

- 3 個 code block（CLAUDE.md 模板/hooks 配置/skills 結構）
- 「95% 用戶」的說法是估算，非實測數據

### 批判性觀察

- **「95%」缺乏數據支撐**：這是修辭性說法而非實測數字
- **內容與其他 harness 文章重疊度高**：核心觀點在多篇文章中重複出現，增量價值有限
- **3 個 code block 較少**：考慮文章主題（harness 設計），示例略顯不足
