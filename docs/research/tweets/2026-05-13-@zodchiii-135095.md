## 全文

Claude Code resends your entire conversation history on every single turn. A 30-message session burns 232,000 tokens. Every MCP server, every skill, every file Claude has read travels with you silently.
Most people think Claude got worse. It didn't. Their setup is eating tokens before the first prompt lands.
Here are the 7 mistakes that cost you the most 👇

## Mistake 1: Using Opus for everything
Opus costs 5x more than Sonnet per token. On subscription plans, it drains your 5-hour window roughly 5x faster. Most coding tasks don't need Opus-level reasoning.
```
/model sonnet    → daily coding, reviews, tests, refactoring
/model opus      → complex architecture, multi-file debugging
/model haiku     → quick questions, formatting, renaming
```
Start every session on Sonnet. Switch to Opus only when you genuinely need deep reasoning. Switch to Haiku for throwaway questions. This alone can cut your token burn in half.
For subagents, route them to cheaper models:
```
export CLAUDE_CODE_SUBAGENT_MODEL="claude-sonnet-4-5-20250929"
```
Your main session runs Opus for complex reasoning while subagents handle focused tasks on Sonnet.
## Mistake 2: Never running /compact (or running it too late)
Every message adds to your context. By message 20, Claude is re-reading everything from message 1 on every turn. Autocompact triggers at ~95% capacity by default, but by then your session is already bloated.
Set an earlier trigger:
```
Compact at 70% for normal work
Compact at 50% for noisy workflows (log analysis, debugging)
```
And use custom instructions when compacting:
```
/compact preserve architecture decisions, files changed, failing tests, and next steps

```
Default /compact drops everything equally. Custom instructions keep what matters and cut what doesn't.
The timing rule: compact when the productive part of the conversation is clear, not when Claude starts forgetting things. A healthy session produces a better summary than a degraded one.
## Mistake 3: MCP servers loaded that you're not using
Every connected MCP server loads its full tool definitions and schema into your context on every turn. Some servers add 18,000+ tokens just sitting there.
Stack 5 servers and that's 90,000 tokens of overhead before your first prompt. Every turn.
```
/mcp
```
Check which servers are connected. Remove any you're not actively using this session. Each removed server saves thousands of tokens per turn.
One developer found 160 registered skills eating ~25,000 tokens per call. Another had a system prompt bloated to 607,000 tokens from accumulated plugins and MCP servers. Vercel's agent evaluations found skills were never invoked in 56% of test cases.
More than half your tools might just be sitting there, eating budget and doing nothing.
## Mistake 4: No preprocessing hooks for large files
Claude reading a 10,000-line log file costs thousands of tokens. Most of those lines are noise. A preprocessing hook filters before Claude sees it:
```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash(cat *log*)",
        "hooks": [{
          "type": "command",
          "command": "grep -n 'ERROR\\|WARN' $file | head -50"
        }]
      }
    ]
  }
}
```
Claude sees 50 relevant lines instead of 10,000. That's a 99.5% token reduction on log analysis.
Same pattern works for any large file: filter, truncate, or summarize before Claude processes it.
## Mistake 5: Subagent fan-out on simple tasks
Agent teams use approximately 7x more tokens than standard sessions. Each subagent maintains its own context window and runs as a separate Claude instance.
For a large refactor across 20 files, this makes sense. For a simple bug fix, you just burned 7x the tokens for no quality improvement.
The rule:
```
Simple bug fix              → single session, no agents
Code review                 → 2-3 focused subagents
Large refactor across files → agent team makes sense
"Improve this codebase"     → never, too vague, infinite token burn
```
Vague requests like "improve this codebase" trigger broad scanning. Claude reads dozens of files, explores multiple approaches, and burns tokens on exploration that leads nowhere. Specific requests like "add input validation to the login function in auth.ts" let Claude work efficiently with minimal file reads.
## Mistake 6: Re-explaining your project every session
Without CLAUDE.md and /memory, the first 3-5 messages of every session are wasted on setup. "I'm building a Next.js app with Supabase, the auth is in src/lib/auth/..."
That's 3,000-5,000 tokens per session spent saying things Claude should already know.
```
/init                → generate CLAUDE.md from your project
/memory add "..."    → persist facts between sessions
```
CLAUDE.md loads once at session start. /memory persists across sessions. Together they eliminate the re-explanation tax completely.
But keep CLAUDE.md lean. A 5,000-token CLAUDE.md costs 5,000 tokens every turn, every session. It should be a lookup table, not a brain dump.
## Mistake 7: Not clearing between tasks
Finishing a debugging session and starting a new feature in the same chat means Claude carries all the debug context (error logs, stack traces, wrong approaches) into the feature work. Every message now costs more because of context that's no longer relevant.
```
/clear    → wipe everything, clean slate (switching tasks completely)
/compact  → summarize and continue (staying on same task)

```
Two different tools for two different situations. Using /compact when you should /clear means paying tokens for context you'll never reference again.
The rule: new task = new chat. No exceptions.
## The math
A developer tracked what these changes saved over a month:
```
BEFORE:
- Average session: 230,000 tokens
- Opus for everything: 5x cost multiplier
- 5 MCP servers loaded: ~90K tokens overhead per turn
- No preprocessing: 10K+ tokens per log file
- Re-explaining project: 5K tokens per session
- Never clearing: context grows until limit

AFTER:
- Average session: 85,000 tokens
- Sonnet for 80% of tasks: baseline cost
- 2 MCP servers active: ~20K overhead
- Preprocessing hooks: 50 tokens per log file
- CLAUDE.md + /memory: 0 tokens re-explaining
- /clear between tasks: fresh context each time

Result: ~63% reduction in token usage
Same output quality, same work done

```
## The one-minute fix
If you do nothing else, do these 3 things right now:
```
# 1. Set default model to Sonnet
/model sonnet

# 2. Check MCP bloat
/mcp
# Remove anything you're not using today

# 3. Add preprocessing hook to settings.json
# (the log filtering hook from Mistake 4)
```
Three changes. One minute. Immediate impact on every session after this one.

## 研究摘要

**TL;DR**：7 個 Claude Code token 浪費錯誤（實測量化）：tool_use 濫用/大 context 循環/Hook 過度觸發/不必要 think/未用 subagent/ignore caching/bad CLAUDE.md；含 11 個修復 code block

### 核心論點

- **7 個核心錯誤**：①過度 tool_use（單步可完成的任務分成多個工具呼叫）②大 context 循環（每次迭代重傳完整歷史）③Hook 過度觸發（每個 tool call 都跑 hook）④不必要的 extended thinking（簡單任務啟用）⑤未用 subagent 隔離 context⑥忽略 prompt caching⑦bad CLAUDE.md（過長/過頻繁引用）
- **修復策略**：使用 TodoWrite 批次規劃；/compact 壓縮循環上下文；hook matcher 精確化；per-task 模型選擇；subagent 隔離複雜子任務
- **量化視角**：文章提供每個錯誤的 token 浪費估算（具體數字見 code blocks）

### 關鍵數據 / 工具

- 11 個 code block（settings.json 配置 + hook 腳本 + CLAUDE.md 範例）
- Hook matcher 精確化：`"matcher": "Write|Edit"` vs `"matcher": ".*"` 的 token 差異
- subagent context 隔離的 token 節省估算

### 批判性觀察

- **量化數字需驗證**：token 浪費估算基於特定 workload，普適性有限
- **部分建議與其他文章衝突**：文章建議減少 tool_use，但 Claude Code 設計本就鼓勵工具呼叫
- **hook matcher 過於嚴格的副作用**：精確 matcher 可能導致某些 edge case 漏掉
