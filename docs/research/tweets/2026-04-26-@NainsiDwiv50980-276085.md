## 全文

I didn't change the prompt.

I changed the repo.

And Claude went from "kind of helpful" to "building like it understands the system."

That's the part no one talks about.

Claude isn't limited by prompts.
It's limited by the environment you drop it into.

Most people keep tweaking instructions.

But the real upgrade looks like this:

- CLAUDE.md → defines how it thinks
- skills → turn tasks into repeatable execution
- hooks → enforce what can't break
- docs → give it structured context

Same model. Same inputs.

Completely different behavior.

At some point you realize:

You're not prompting an AI…

You're designing the system it operates in.

Bookmark it for later

## 研究摘要

**TL;DR**：Claude 的行為差距來自 repo 環境設計（CLAUDE.md、skills、hooks、docs），而非 prompt 調整——同一模型、同樣輸入，結果天差地別。

### 核心論點

1. Claude 的限制不在 prompt 品質，而在「它被放入的環境」——repo 結構決定行為。
2. 四個關鍵環境元素：CLAUDE.md（思維定義）、skills（可重複執行單元）、hooks（強制執行不可破壞的規則）、docs（結構化 context）。
3. 核心洞見：使用 AI 的本質是「設計系統讓 AI 在其中運作」，而非「持續調整指令」。

### 關鍵數據 / 工具

- 四層環境：CLAUDE.md、skills、hooks、docs
- 對比：「kind of helpful」vs「building like it understands the system」

### 批判性觀察

1. **更精煉的摘要版本**：相較於同日 @Suryanshti777 的 8 層詳細版，此推文是更易傳播的精簡版——「改 repo 不改 prompt」的敘事框架更強，但資訊量更少。
2. **無新觀點**：四個元素（CLAUDE.md、skills、hooks、docs）是 Claude Code 標準配置，無創新性。
3. **對工程師的價值**：對非技術背景的「vibe coder」有引導作用，對已有完整架構的工程師無實質幫助。
