# Project 02：設計你的 Harness

## 專案說明

本專案帶你設計並實作一個有 Planner / Generator / Evaluator 三層架構的 Harness。你將把一個真實的模糊需求轉換成可執行 spec，讓 Generator 逐項實作，並用外部工具（非模型自評）驗證結果。

**完成後的成果**：
- 可執行的 Planner 提示詞（將模糊需求分解成 spec items）
- 安全 PreToolUse Hook（阻斷危險指令 + 保護敏感檔案）
- Evaluator 外部驗證腳本（使用 pytest / mypy / playwright）
- 完整的三層 Harness 文件化設計

## 任務清單

### 階段一：設計 Planner

- [ ] **任務 1.1**：選一個「模糊需求」（例如：「加一個用戶後台管理系統」或「優化應用程式效能」）
- [ ] **任務 1.2**：撰寫 Planner 提示詞，要求 Claude 將模糊需求分解成：
  - 具體的 spec items（每個 item 獨立可執行）
  - 每個 item 的驗收命令
  - 預估完成順序和依賴關係
- [ ] **任務 1.3**：執行 Planner，評估輸出的 spec 是否足夠細粒度（每個 item 不超過 2 小時完成）
- [ ] **任務 1.4**：如果 spec items 太粗，繼續拆分直到每個 item 有明確的「驗收命令」

### 階段二：設計 Generator 邊界

- [ ] **任務 2.1**：在 CLAUDE.md 中加入 Generator 的邊界規則：
  - 每次只實作一個 spec item
  - 完成一個 item 後必須更新 feature_list.json 的狀態
  - Context 快滿時應停下，而不是跳過驗證
- [ ] **任務 2.2**：建立 feature_list.json（從 Planner 輸出的 spec 生成）
- [ ] **任務 2.3**：實作一個 Generator 執行流程（可以是 bash 腳本或手動流程）：
  - 讀取 feature_list.json 中第一個 `pending` item
  - 執行 Generator（實作這個 item）
  - 執行驗收命令

### 階段三：設計 Evaluator

- [ ] **任務 3.1**：建立 `scripts/evaluate.sh`，包含：
  - 單元測試（pytest 或 npm test）
  - 類型檢查（mypy 或 tsc）
  - Lint（ruff 或 eslint）
  - 如果有 UI：E2E 測試（playwright 或 cypress）
- [ ] **任務 3.2**：設定 Evaluator 的「Hard Threshold」：任意一項失敗 → 整個評估失敗
- [ ] **任務 3.3**：在 CLAUDE.md 中加入：Generator 完成後**必須**執行 Evaluator，且**不允許**自行宣布通過

### 階段四：設計 PreToolUse Security Hook

- [ ] **任務 4.1**：建立 `.claude/hooks/security-check.sh`（PreToolUse hook），包含：
  - 阻斷危險 bash 指令（rm -rf /、DROP TABLE 等）
  - 阻止讀取敏感檔案（.env、.pem、.key、~/.ssh 等）
  - 阻止寫入受保護路徑
- [ ] **任務 4.2**：在 `.claude/settings.json` 中設定 PreToolUse hook
- [ ] **任務 4.3**：測試 hook：嘗試讓 Claude 執行一個被阻斷的指令，確認 hook 正常工作

### 階段五：整合與測試

- [ ] **任務 5.1**：用你設計的 Harness 完成一個完整的 Planner → Generator → Evaluator 循環
- [ ] **任務 5.2**：記錄這個循環中：
  - Planner 分解了多少個 spec items？
  - Generator 在哪些地方需要重試？
  - Evaluator 發現了多少個 Generator 沒有發現的問題？
- [ ] **任務 5.3**：把失敗的案例加入 CLAUDE.md 的「禁止事項」（Ratchet 原則）

## 參考實作

### Planner 提示詞模板

```
你是一個技術 Planner。你的任務是把以下模糊需求分解成具體的、可執行的 spec items。

需求：[貼入你的模糊需求]

請輸出一個 JSON 格式的 spec，包含：
1. 每個 spec item 的具體描述（一句話，動詞開頭）
2. 每個 spec item 的驗收命令（可以直接執行的 bash 命令）
3. 每個 spec item 的預估工時（分鐘）
4. 依賴關係（哪些 items 必須在此 item 之前完成）

規則：
- 每個 spec item 的預估工時不超過 120 分鐘
- 每個 spec item 必須有可執行的驗收命令（不接受「手動檢查」）
- 依賴關係要明確（避免循環依賴）
```

### scripts/evaluate.sh（Evaluator 腳本）

```bash
#!/bin/bash
# Evaluator：外部驗證腳本
# 任何一個步驟失敗 → 整體評估失敗
set -euo pipefail

FEATURE_ID="${1:-all}"
EXIT_CODE=0
REPORT=""

echo "=========================================="
echo "Evaluator: Starting validation"
echo "Feature: $FEATURE_ID"
echo "=========================================="

# 1. 單元測試
echo ""
echo "Step 1: Running unit tests..."
if pytest tests/ -q --tb=short 2>&1; then
  REPORT="${REPORT}\n✅ Unit tests: PASSED"
else
  REPORT="${REPORT}\n❌ Unit tests: FAILED"
  EXIT_CODE=1
fi

# 2. 類型檢查
echo ""
echo "Step 2: Type checking..."
if python -m mypy src/ --strict -q 2>&1; then
  REPORT="${REPORT}\n✅ Type check: PASSED"
else
  REPORT="${REPORT}\n❌ Type check: FAILED"
  EXIT_CODE=1
fi

# 3. Lint
echo ""
echo "Step 3: Linting..."
if ruff check src/ 2>&1; then
  REPORT="${REPORT}\n✅ Lint: PASSED"
else
  REPORT="${REPORT}\n❌ Lint: FAILED"
  EXIT_CODE=1
fi

# 結果報告
echo ""
echo "=========================================="
echo "Evaluation Report:"
echo -e "$REPORT"
echo "=========================================="

if [ $EXIT_CODE -eq 0 ]; then
  echo "✅ EVALUATION PASSED - Feature can be marked as done"
else
  echo "❌ EVALUATION FAILED - Do not mark feature as done"
fi

exit $EXIT_CODE
```

### .claude/hooks/security-check.sh（完整版）

```bash
#!/bin/bash
# PreToolUse Security Hook
set -euo pipefail

INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name')
TOOL_INPUT=$(echo "$INPUT" | jq -r '.tool_input')

# --- 1. Bash 指令安全檢查 ---
if [ "$TOOL_NAME" = "bash" ]; then
  COMMAND=$(echo "$TOOL_INPUT" | jq -r '.command // empty')
  
  declare -a DANGEROUS_PATTERNS=(
    "rm -rf /"
    "rm -rf ~"
    "DROP TABLE"
    "DROP DATABASE"
    "DROP SCHEMA"
    "terraform destroy"
    "git push --force.*main"
    "git push --force.*master"
    "git push -f.*main"
    "git push -f.*master"
    "> /dev/sda"
    "dd if=/dev/zero"
    "chmod -R 777 /"
    "chown -R.*/"
  )
  
  for pattern in "${DANGEROUS_PATTERNS[@]}"; do
    if echo "$COMMAND" | grep -qiE "$pattern"; then
      echo "SECURITY BLOCK: dangerous pattern detected" >&2
      echo "Pattern: $pattern" >&2
      echo "Command: $COMMAND" >&2
      exit 2
    fi
  done
fi

# --- 2. 檔案讀取安全檢查 ---
if [ "$TOOL_NAME" = "read_file" ]; then
  FILE_PATH=$(echo "$TOOL_INPUT" | jq -r '.path // empty')
  
  declare -a SENSITIVE_PATTERNS=(
    "^\.env$"
    "\.env\."
    "\.pem$"
    "\.key$"
    "^id_rsa$"
    "credentials\.json$"
    "service-account.*\.json$"
    "\.aws/credentials"
    "\.ssh/"
    "\.gnupg/"
  )
  
  for pattern in "${SENSITIVE_PATTERNS[@]}"; do
    if echo "$FILE_PATH" | grep -qE "$pattern"; then
      echo "SECURITY BLOCK: reading sensitive file" >&2
      echo "File: $FILE_PATH" >&2
      exit 2
    fi
  done
fi

# --- 3. 檔案寫入安全檢查 ---
if [ "$TOOL_NAME" = "write_file" ] || [ "$TOOL_NAME" = "edit_file" ]; then
  FILE_PATH=$(echo "$TOOL_INPUT" | jq -r '.path // empty')
  
  declare -a PROTECTED_PATHS=(
    "\.env$"
    "\.env\."
    "\.pem$"
    "\.key$"
    "^\.git/"
    "\.github/workflows/"
    "\.claude/settings\.json$"
  )
  
  for pattern in "${PROTECTED_PATHS[@]}"; do
    if echo "$FILE_PATH" | grep -qE "$pattern"; then
      echo "SECURITY BLOCK: writing to protected path" >&2
      echo "File: $FILE_PATH" >&2
      echo "This file requires explicit human approval before modification" >&2
      exit 2
    fi
  done
fi

exit 0
```

### CLAUDE.md Generator 邊界規則

```markdown
## Generator 邊界規則（嚴格遵守）

1. **每次只實作一個 spec item**：從 feature_list.json 中取第一個 `pending` item 開始
2. **完成後立即更新狀態**：執行驗收命令通過後，更新 feature_list.json 的 `status` 為 `done`
3. **不允許自行宣布通過**：完成實作後，必須執行 `bash scripts/evaluate.sh [feature-id]`，讓 Evaluator 驗證
4. **Context 快滿時停下**：不要在 context 接近上限時強行完成，應停下並在 claude-progress.md 記錄當前狀態
5. **失敗即停止**：Evaluator 報告失敗時，不要繼續下一個 spec item，先修復當前的失敗
```

## 驗收標準

完成本專案後，你應該能夠：

- **驗收 1**：給 Claude 一個模糊需求（3-5 個字），Planner 能輸出至少 5 個具體的、有驗收命令的 spec items
- **驗收 2**：執行 PreToolUse hook，阻止一個危險指令（例如：`rm -rf /tmp/test`）並顯示清晰的錯誤訊息
- **驗收 3**：Evaluator 腳本在測試失敗時返回非 0 exit code，成功時返回 0
- **驗收 4**：完成一個完整的 Planner → Generator → Evaluator 循環，Evaluator 發現至少一個 Generator 「宣布完成」但實際有問題的案例
- **驗收 5**：整個流程的文件化（CLAUDE.md 中有明確的 Generator 邊界規則 + Evaluator 的使用說明）
