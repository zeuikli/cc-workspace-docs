## 全文

Claude Code resends your entire conversation history on every single turn. A 30-message session burns 232,000 tokens. Every MCP server, every skill, every hook fires before your first word lands.

Most people think Claude got worse. It didn't. Their setup is eating tokens before the first prompt lands.

Here are the 7 mistakes that cost you the most:

## Mistake 1: Using Opus for everything

Opus costs 5x more than Sonnet per token. On subscription plans, it drains your 5-hour window roughly 5x faster. Most coding tasks don't need Opus-level reasoning.

Start every session on Sonnet. Switch to Opus only when you genuinely need deep reasoning. Switch to Haiku for throwaway questions. This alone can cut your effective token spend significantly.

For subagents, route them to cheaper models. Your main session runs Opus for complex reasoning while subagents handle focused tasks on Sonnet.

## Mistake 2: Never running /compact (or running it too late)

Every message adds to your context. By message 20, Claude is re-reading everything from message 1 on every turn. Autocompact triggers at ~95% capacity — that's too late.

Set an earlier trigger and use custom instructions when compacting. Default /compact drops everything equally. Custom instructions keep what matters and cut what doesn't.

The timing rule: compact when the productive part of the conversation is clear, not when Claude starts forgetting things.

## Mistake 3: MCP servers loaded that you're not using

Every connected MCP server loads its full tool definitions and schema into your context on every turn. Some servers add 18,000+ tokens just sitting there idle.

Stack 5 servers and that's 90,000 tokens of overhead before your first prompt. Every turn.

Check which servers are connected. Remove any you're not actively using this session. One developer found 160 registered skills eating ~25,000 tokens per call. Another had a system prompt bloated to 607,000 tokens from accumulated plugins.

## Mistake 4: No preprocessing hooks for large files

Claude reading a 10,000-line log file costs thousands of tokens. Most of those lines are noise. A preprocessing hook filters before Claude sees it — Claude sees 50 relevant lines instead of 10,000. That's a 99.5% token reduction on log analysis.

Same pattern works for any large file: filter, truncate, or summarize before Claude processes it.

## Mistake 5: Subagent fan-out on simple tasks

Agent teams use approximately 7x more tokens than standard sessions. Each subagent maintains its own context window and runs as a separate Claude instance.

For a large refactor across 20 files, this makes sense. For a simple bug fix, you just burned 7x the tokens for no quality improvement.

Vague requests like "improve this codebase" trigger broad scanning. Claude reads dozens of files, explores multiple approaches, and burns tokens on exploration.

## Mistake 6: Re-explaining your project every session

Without CLAUDE.md and /memory, the first 3-5 messages of every session are wasted on setup. That's 3,000-5,000 tokens per session spent saying things Claude should already know.

CLAUDE.md loads once at session start. /memory persists across sessions. Together they eliminate the re-explanation tax completely.

But keep CLAUDE.md lean. A 5,000-token CLAUDE.md costs 5,000 tokens every turn, every session. It should be a lookup table, not a brain dump.

## Mistake 7: Not clearing between tasks

Finishing a debugging session and starting a new feature in the same chat means Claude carries all the debug context — error logs, stack traces, wrong attempts — into the new task.

Two different tools for two different situations. Using /compact when you should /clear means paying tokens for context you'll never reference again.

The rule: new task = new chat. No exceptions.

## The one-minute fix

If you do nothing else, do these 3 things right now: audit your active MCP servers, set a /compact trigger below 95%, and start new tasks in new chats. Three changes. One minute. Immediate impact on every session after this one.

## 研究摘要

**TL;DR**：Claude Code 上下文設定的 7 個 token 浪費陷阱，每條附 30 秒修復方案

### 核心論點

1. **模型選擇策略**：Opus 費用是 Sonnet 的 5 倍，日常任務用 Sonnet；subagent 路由至更便宜模型
2. **Compact 時機**：應在 70% 前主動執行，不要等到 95% autocompact 觸發
3. **MCP/Skills 瘦身**：每個閒置 MCP server 每輪注入 18,000+ tokens；160 個 skills 一次花費 25,000 tokens
4. **預處理 hooks**：大檔案（10K 行日誌）在餵給 Claude 前先過濾，可達 99.5% token 減少
5. **CLAUDE.md 精簡**：保持在 1,500 tokens 以下；長度過大每輪都是固定稅

### 關鍵數據 / 工具

- 30 訊息 session = 232,000 tokens（歷史重讀成本）
- Subagent fan-out 消耗約 7x tokens
- MCP server idle overhead：18,000+ tokens/turn
- CLAUDE.md 目標：< 1,200 words（~1,500 tokens）
- 工具：`/compact`、`/clear`、`/memory`、`CLAUDE.md`

### 批判性觀察

1. **量化數字缺來源**：「232,000 tokens」等數字未說明計算方式，可能因設定不同而差異很大
2. **「new task = new chat」過於絕對**：有些跨任務共享的上下文（如架構決策）清除反而增加重新建立成本
3. **實際節省難以複現**：每個技巧的「節省 X%」假設前提不同，組合效果不會簡單相加
