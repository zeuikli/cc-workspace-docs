## 一、把 Claude 当 junior 同事，不是态度问题，是 prompt 工程

很多人看到这条会以为是「对它说话礼貌一点」。其实跟礼貌没关系，这是计算路径的选择。

LLM 接收命令式 prompt 时，触发的是「按指令执行」的浅层模式，直接 pattern match，输出快但思考浅。接收描述式 prompt 时，触发的是「理解后再行动」的深层模式，会展开 chain of thought，输出慢但准确度高一截。

同一个 bug，「修一下这里」和「这里看起来怪怪的，你查查为啥」出来的代码质量能差一倍。前者它就直接改你指出的那行，后者它会先去理解上下文，可能发现真正的问题在另一个地方。

更深一层，描述要具体且不带预设答案。「这里有 race condition」会让它只查 race condition，「这里偶尔会出错你查查」会让它展开完整排查。你 prompt 里预设的答案越少，激发的推理就越多。

## 二、60% 用 /compact，是对抗 lost in the middle，不是省 token

这个 60% 经常被解读成「省 token 的阈值」。其实不是。

Stanford 2023 年有篇论文揭示了 lost in the middle 现象，LLM 在长 context 里的注意力是 U 型曲线，开头记得最牢，结尾记得最牢，中间最容易被忽略。

60% 这条线不是省钱的红线，是注意力开始衰减的经验线。超过这个点之后，你前面提过的关键需求会开始被「视而不见」，模型会犯一些「明明你说过了」的错。

/compact 真正的价值是它会保留关键决策和未完成事项，把中间的散乱讨论压掉。这不是粗暴截断，是在做注意力的重新聚焦。

判断什么时候该 compact，不要光看 token 数，要看模型是不是开始忘事。一旦它问出「你想做什么」之类的问题，立刻 compact，迟一步它就会带着错误前提继续往下做。

## 三、跑偏了 exit 重开，是 token 经济学，不是耐心问题

很多人遇到 Claude 走偏，第一反应是「我再 prompt 它一下让它修正」。这个反应是错的。

被污染的 context 里，你写的修正 prompt 自己占 context，它的道歉占 context，它修正后的输出还会被错误推理影响。人类有 confirmation bias，LLM 也有，被前面的错误锚定之后很难真正跳出来。

总成本比重开高 2 到 3 倍，最终输出还可能更差。

应该建立的反射是 fail fast，看到走偏立刻 ESC，重新组织思路再 prompt，比让它自己掰回来快得多。重开不是失败，是更聪明的资源分配。

这个反射很多人建立不起来，因为觉得「重开是浪费」。其实越往后修，浪费越大。

## 四、sub agents 真正的价值是上下文隔离，不是并行

并行只是表象，隔离才是核心。

sub agents 把 context 从一维资源变成了树形资源。父 agent 是指挥者，看大局，不需要知道每个细节怎么实现。sub agent 是执行者，可以激进探索，包括各种失败和绕路。父 agent 只看 sub agent 的最终结论，那些失败的探索不会污染主线。

什么任务最适合开 sub agent，是「需要大量探索但只需要少量结论」的任务。比如「在 codebase 里找所有用了某个 deprecated API 的地方」，让 sub agent 把代码翻完，回来交一份清单就行。父 agent 不需要知道它翻过哪些文件，更不需要被它翻过的代码污染主上下文。

理解了这一点，你会发现很多任务都可以这样拆。「调研一下 X 库怎么用」「查一下这个 bug 在哪几个文件出现过」「列出所有需要改的测试」，全都是典型的 sub agent 任务。

## 五、简单活给 Haiku 不只是省钱，是避免「过度设计」

这背后是大模型的过度对齐问题。

大模型训练时见过太多复杂场景，对简单任务也会本能地加边缘情况处理，加错误处理，加注释和文档，用更复杂的设计模式。

写一个简单的 helper 函数，Sonnet 可能给你 30 行，Haiku 给你 10 行。后者在大多数场景反而更对。

我自己的判断标准是任务的「认知步骤数」：

1 到 2 步的活丢 Haiku 5 步以上留给 Sonnet 或 Opus 10 步以上才开 ultrathink

不要无脑用最强模型，最强模型对简单任务有「设计过度」的副作用，输出的代码后期还要花时间删冗余。

## 六、/loop 挂 3 天的真正用法是 monitoring，不是写代码

很多人看到 /loop 第一反应是「让 Claude 连续写 3 天代码」，其实不是。

/loop 真正的用法是把 Claude 从同步工具变成异步同事。它在那挂着，定期检查状态，有事叫你。

好的 /loop 任务有几个特征：

输入是状态，不是动作 输出是「有事/没事」的判断 处理时间不连续，可以等

比如每 10 分钟看一下 CI 是不是过了，没过分析失败原因，每小时扫一下数据库慢查询，每天看一遍 PR review 队列。这些以前是「我得记得去看」的事，现在变成「Claude 看到再叫我」。

这是 Claude 从写代码工具升级成值班同事的关键。

## 把这些串起来看

如果只看 32 个技巧，你最多能复述操作。但如果把上面这 6 个点连起来看，你会发现 Nate Herk 真正的心法，是一个心智模型：

把 Claude Code 当成一个「有限注意力，有限时间，有性格」的同事来管理，不要当成「无限算力的工具」来用。

所有技巧背后都是这个心智模型在起作用：

注意力有限，所以要做 context 管理，对应的是 /compact，/clear，sub agent 隔离 时间有限，所以要异步化，对应的是 /loop，agent team，多 session 并行 有性格，所以要 prompt 沟通，对应的是 plan mode，让它反问，junior 同事心态

心智模型一旦换过来，剩下的技巧都是延伸，自己也能想出来。

我自己用 Claude Code 写代码差不多大半年了，最大的认知升级不是某个具体技巧，而是从「我用一个工具」变成「我管一个同事」。这两种心态出来的代码质量，效率，token 消耗，差着一个数量级。

## 研究摘要

**TL;DR**：32 個技巧只是表象，核心是「把 Claude 當有限注意力/時間/性格的同事管理」這個心智模型。

### 核心論點

1. **描述式 prompt > 命令式 prompt**：命令式觸發 pattern match 淺層模式；描述式（不預設答案）觸發 chain of thought 深層模式，準確度差一倍。
2. **60% compact 是對抗 Lost in the Middle**：根據 Stanford 2023 論文，注意力呈 U 型曲線，60% 是注意力衰減的經驗線而非省 token 紅線。看到模型「忘事」立刻 compact。
3. **Fail fast 重開比修正便宜 2-3 倍**：被污染的 context 中，修正 prompt 自身、模型道歉、錯誤錨定都繼續吃 token，重開反而是更聰明的資源分配。
4. **Sub agent 真正價值是 context 隔離**：把 context 從一維資源變成樹形資源；父 agent 只看結論，sub agent 的探索失敗不污染主線。適合「大量探索 + 少量結論」任務。
5. **Haiku 防止 over-design**：大模型過度對齊讓簡單任務也加邊界處理/錯誤處理/設計模式。判斷標準：認知步驟 1-2 步 Haiku、5+ 步 Sonnet/Opus、10+ 步才開 ultrathink。
6. **/loop 是 monitoring，不是寫程式**：把 Claude 從同步工具變異步同事。好任務特徵：輸入狀態（非動作）、輸出有事/沒事判斷、處理時間不連續。
7. **統一心智模型**：注意力有限 → context 管理；時間有限 → 異步化；有性格 → prompt 溝通。

### 關鍵數據 / 工具

- **60%** context 使用率：注意力衰減經驗線
- **2-3x**：context 污染後修正的 token 成本倍數
- **Stanford 2023** Lost in the Middle 論文：U 型注意力曲線
- **Sonnet 30 行 vs Haiku 10 行**：相同 helper 函數的 over-design 對比
- 認知步驟分檔：**1-2 / 5+ / 10+** → Haiku / Sonnet+Opus / ultrathink
- /loop 範例：每 10 分鐘檢查 CI、每小時掃慢查詢、每天看 PR queue

### 批判性觀察

1. **論點層次清晰但證據以「現象描述」為主**：除 Stanford 論文外，2-3x 成本倍數、Sonnet vs Haiku 行數對比皆為作者經驗值，非實測數據。對 production 決策可作參考但需自驗證。
2. **與既有規則高度互補但有重疊**：本 workspace 的 `context-management.md` 已採 30-35% 主動 compact 標準（更激進），與本文 60% 不衝突 — 60% 是「最遲」、30-35% 是「最佳」。三、四、五點原理可作為現有規則的「為什麼」補充。
3. **「junior 同事心態」對應本 workspace 的 `Think Before Coding`**：作者強調「描述式、不預設答案」的 prompt 觸發深度推理，這與 `core.md` 中「實作前假設顯露」屬同一現象的不同切面（一個從用戶端、一個從 Claude 端）。
