來源: https://x.com/eng_khairallah1/status/2046968076101673250作者: Khairallah AL-Awady (@eng_khairallah1) 發布日期: 2026-04-22 收錄日期: 2026-05-17 時長: 14:24 原始推文描述: 🚨 Anthropic's own engineer just dropped 14 minutes on building agents the right way. most people will watch it try to build one it won't work well. not because the video was wrong. because their prompts are wrong. watch the video. then read the guide.

核心摘要 ​

Anthropic 工程師 Barry 在 AI Engineer Summit 的 14 分鐘演講，提出三個打造有效 Agent 的核心原則。

1. 不要什麼都用 Agent（Don't Build Agents for Everything） ​

Agent 是用來擴展複雜且高價值任務的工具，不是每個 use case 的升級選項。決策清單：

考量判斷方向
任務複雜度決策樹容易畫出 → 改用 workflow，明確控制每個節點
任務價值每次任務預算 ~$0.10 → tokens 有限，用 workflow 覆蓋主流情境即可
關鍵能力風險確認沒有重大瓶頸（寫程式→debug→從錯誤恢復）；有瓶頸 → 縮小範圍
錯誤代價高風險且難發現 → 先設 read-only、加 human-in-the-loop，限制自主程度

為何 Coding Agent 適合：任務模糊且高價值、模型能力已驗證、輸出可透過 unit test / CI 機械性驗證。

2. 保持簡單（Keep It Simple） ​

Agent 的本質 = Model + Tools in a loop，只有三個核心元件：

  • Environment：Agent 運作的系統
  • Tools：Agent 取得回饋、採取行動的介面
  • System Prompt：定義目標、限制、理想行為

任何前期複雜度都會拖慢迭代速度。先把這三個元件做好，再優化（cache trajectory、平行化 tool call、呈現進度建立用戶信任）。

3. 像 Agent 一樣思考（Think Like Your Agents） ​

每個推理步驟，模型看到的只有 10–20K tokens 的有限上下文。建議：

  • 親身體驗：把自己放進 agent 的 context window，確認資訊是否充足且一致
  • 問 Claude 自己：把 system prompt / tool descriptions 丟給 Claude，問「有沒有歧義？你能遵循嗎？」
  • 分析軌跡：把整個 agent trajectory 丟給 Claude，問「為什麼在這裡做這個決定？」
  • 實驗發現：agent 需要知道螢幕解析度才能正確點擊、需要推薦行動與限制作為護欄

個人展望（Open Questions） ​

  1. Budget-aware Agent：目前對 Agent 的 cost/latency 缺乏有效控制；需要定義如何用 time/money/tokens 設定並執行預算
  2. Self-evolving Tools：Agent 應能自行設計並改進 tool 描述，讓工具更符合各 use case 的 ergonomics
  3. Multi-Agent 協作：預測年底前會在 production 看到更多 multi-agent；關鍵挑戰是如何從同步 user-assistant turns 擴展到異步通訊模型
