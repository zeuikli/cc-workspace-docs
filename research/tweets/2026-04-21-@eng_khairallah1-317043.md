## 全文

I have been using Claude Code daily for months. These are the 35 techniques that make it feel like a cheat code.

### Essential Commands (01 to 08)

01. **Plan Mode (Shift + Tab)**：切換到 Plan Mode 分析 codebase 並創建架構計劃，不寫任何代碼。先審閱計劃、批准後再切換回實作模式。「This single habit prevents more bugs than any other technique on this list.」

02. **/compact**：30-45 分鐘後 context 膨脹，壓縮為關鍵決策和當前狀態的摘要

03. **/clear**：開始新任務時清空，防止不同 feature 的 context 衝突

04. **/init**：在新專案開始時運行，掃描 codebase 自動生成 CLAUDE.md（含結構、技術棧、編碼模式、架構決策）

05. **/cost**：查看 session 的 token 用量。「Check this every hour during long sessions.」

06. **/memory**：添加跨 session 持久的指令（「Always use TypeScript strict mode」「Always run tests after modifying any file in /src/core」）

07. **! prefix**：用 ! 前綴直接執行 terminal 命令（不離開 Claude Code 界面運行測試/git 等）

08. **Multi-Model Switching**：Opus 用於規劃和架構決策；Sonnet 用於實作執行。「Plan with the thinker. Build with the builder.」

### Productivity Techniques (09 to 18)

09. **The Reference File Technique**：「Look at how authentication is implemented in src/auth/login.ts. Implement password reset following the exact same patterns.」比口頭描述產生更一致的代碼。

10. **The Screenshot Debug**：截圖貼上 (Ctrl+V) + 簡短描述比長篇文字描述更快更精確。

11. **The Test-First Workflow**：先寫測試（覆蓋 normal/edge/error cases），再實作通過測試的函式。

12. **The Incremental Build**：不要說「build the entire feature」，而是分步驟（schema → API → validation → frontend），每步測試，產出品質遠優於一次性 prompt。

13. **The Codebase Question**：改動前先問「Read src/services/ and explain how data flows...」理解架構再建構，防止架構錯誤。

14. **The Diff Review**：改動後要求「Show me a diff of every file you modified. Explain each change in one sentence.」捕捉非預期修改。

15. **The Error Paste**：完整貼上錯誤信息（不是摘要），加上「Diagnose the root cause step by step before suggesting a fix.」

16. **The Undo Checkpoint**：每次重大改動前 `git add . && git commit -m "checkpoint before [change]"`

17. **The Parallel Session**：大型 feature 開兩個 terminal window，各自處理 backend 和 frontend，各自有乾淨 context。

18. **The Documentation Pass**：功能完成後立即生成文件（此時最準確）

### Architecture Techniques (19 to 26)

19. **The Pattern Extraction**：讓 Claude Code 分析現有代碼模式，再用此分析指導新開發

20. **The Interface-First Design**：先定義 interfaces 和 types，再實作

21. **The Error Boundary Mapping**：識別所有可能失敗點，設計錯誤處理策略

22. **The Dependency Audit**：在引入新 dependency 前讓 Claude 評估替代方案

23. **The Refactor Protocol**：重構前先讓 Claude 寫充分的 tests，再執行重構

24. **The Migration Plan**：大型遷移前讓 Claude 生成逐步計劃（帶 rollback 點）

25. **The Security Review**：功能完成後要求安全審查（注入攻擊、認證問題、數據泄露）

26. **The Performance Audit**：讓 Claude 分析 bottlenecks 並提供帶成本估算的修復建議

### Workflow Automation (27 to 31)

27-31. Scripts that run automatically on triggers; Claude-generated make targets; CI/CD hooks; PR description generators; automated testing reporters

### Debug and Recovery (32 to 35)

32. **The Context Rebuild**：Lost context 後快速重建—讓 Claude 讀取相關文件並摘要當前狀態

33. **The Regression Hunter**：「What changed in the last 5 commits that could cause this bug?」

34. **The Clean Slate Debug**：/clear 後重新陳述問題，排除 context 污染的影響

35. **The Step-by-Step Trace**：「Walk through the code execution for this input step by step, predicting the output at each line.」

### The Setup That Ties It All Together

強健的 CLAUDE.md + /memory 的 persistent rules + 一致的 Plan → Build → Test → Document 循環

## 研究摘要

**TL;DR**：35 個 Claude Code 實戰技巧，最高槓桿的是 Plan Mode (Shift+Tab)、Reference File Technique（指向現有代碼而非口頭描述）、Diff Review（捕捉非預期修改）和 Incremental Build 工作流。

### 核心論點

- Plan Mode + Incremental Build 組合是最高 bug 預防槓桿（先計劃後實作，每步測試）
- Reference File Technique 替代口頭描述：讓 Claude 讀現有實作並複製模式，比描述更精確
- Diff Review 是被低估的重要習慣：Claude 有時「善意地」修改你沒要求修改的文件
- Multi-Model Switching（Opus 規劃 + Sonnet 執行）是成本與品質的最優平衡
