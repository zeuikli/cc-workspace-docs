# Karpathy's CLAUDE.md Skills File: The Complete Guide

## 核心摘要

本文詳細介紹了開發者 Forrest Chang 在 GitHub 上創建的 `andrej-karpathy-skills` 項目，該項目根據 Andrej Karpathy 的觀察，將 AI 編程代理常見的失誤轉化為四項行為準則。

**背景與起源**

該項目於 2026 年 1 月 27 日創建，儘管只包含單個配置文件，卻在數週內獲得數万顆星，成為 GitHub 增長最快的項目之一。其成功源於 Karpathy 對大型語言模型編程行為的深入觀察，特別是 AI 編碼代理如何像「自信的初級開發者」一樣行動——快速但往往缺乏謹慎。

**識別的四大問題**

Karpathy 指出了 LLM 編碼的四個關鍵失誤：隱藏假設與混亂管理、過度工程設計與代碼膨脹、無關的副作用編輯、以及缺乏明確的成功標準。這些問題在實際應用中表現為 AI 默默做出假設後執行、構建不必要的抽象層、修改無關代碼的格式與樣式，以及沒有驗證標準的盲目實現。

**四項核心原則**

首先，「編碼前思考」要求 AI 明確陳述假設、呈現多種解釋、提出更簡單的方案，並在困惑時停止。其次，「優先考慮簡潔性」堅持最小化代碼、避免投機性功能、拒絕不必要的抽象層。第三，「精準變更」限制編輯範圍，禁止改進無關代碼或風格漂移。最後，「目標驅動執行」將任務轉化為可驗證的目標，例如撰寫測試後修復代碼。

**CLAUDE.md 文件**

該文件是一個項目記憶卡片，AI 在每個會話開始時自動讀取，提供持久化的上下文。整個文件不超過 70 行，設計上刻意簡潔，以便不佔用代理的上下文空間。開發者可在項目根目錄中使用該文件，或通過包含項目特定指令來自定義。

**實際應用與社區反應**

Reddit 社區廣泛認可這些準則，將 Claude Code 描述為才華橫溢但不可靠的「初級開發者」，需要明確的護欄。社區對話表明，AI 生成代碼的質量與使用者的工程判斷和「上下文工程」技能成正比。此外，AI 已將代碼編寫的瓶頸從實現轉移到架構理解與評估。

**演進意義**

該項目代表了從「氛圍編碼」到「代理工程」的演變。它不旨在限制 AI 的能力，而是通過明確的原則和邊界來引導 AI 的力量。作為「想法文件」模式的實例，它展示了分享理念而非實現的開源新模式。
