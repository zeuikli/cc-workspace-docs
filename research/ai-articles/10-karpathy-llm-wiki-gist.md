# LLM Wiki: A Knowledge Management Pattern

## 項目概述

**名稱**：LLM Wiki  
**作者**：Andrej Karpathy  
**創建日期**：2026年4月4日  
**形式**：GitHub Gist（可執行的知識管理模式）

## 核心概念

Karpathy 提出了一種不同於傳統 RAG（檢索增強生成）的持久化、自演進的知識基礎構建模式。與其在查詢時檢索原始文檔，LLM 可以持續維護一套結構化的 wiki —— 一個相互連結的 Markdown 檔案集合，隨著每個新來源的加入而逐漸豐富。

**與 RAG 的根本區別**：知識被編譯一次，然後持續維護，而非每次查詢都重新推導。

## 系統架構

系統跨越三個層次運作：

**1. 原始來源層（Raw Sources）**

不可更改的精選文檔集合：
- 文章和研究論文
- 圖像和視覺資料
- 影片和多媒體資源

**2. Wiki 層（The Wiki）**

LLM 生成的 Markdown 頁面集合，包含：
- 內容摘要和綜述
- 實體頁面（主題、人物、概念）
- 交叉引用和語義連結
- 引文和出處標註

**3. 結構層（The Schema）**

配置文檔，定義：
- Wiki 的整體結構和組織方式
- 頁面類型和模板
- 工作流程和更新策略

## 核心操作流程

**攝入操作（Ingest）**

新來源被處理時，LLM：
1. 讀取並理解新資料
2. 提取關鍵資訊
3. 更新 10-15 個相關 wiki 頁面
4. 添加新的交叉引用

**查詢操作（Query）**

回答問題時：
1. 查閱相關 wiki 頁面
2. 提供帶引文的答案
3. 有價值的新答案變成新的 wiki 頁面
4. 強制維護知識的單一真理來源

**檢查操作（Lint）**

定期進行健康檢查：
- 識別矛盾和衝突的陳述
- 查找陳舊或過時的聲明
- 發現孤立的頁面
- 找出缺失的交叉引用

## 導航與索引系統

**index.md（內容導向目錄）**

按類別組織的內容索引，便於按主題瀏覽。

**log.md（時間順序日誌）**

附加式的年代記錄，記錄時間戳和變更，支持解析和追蹤。

## 核心原則（傳統中文總結）

**概念核心**

不同於 RAG 的即時檢索，LLM Wiki 建立持久的、可複合增長的知識結構。知識被編譯一次，然後持續維護，而非每次查詢都重新推導。這提高了效率、一致性和知識的深度。

**人機分工**

人類負責：
- 策劃和篩選來源
- 探索和提出問題
- 驗證知識品質

LLM 執行：
- 摘要和提取
- 交叉引用和連結
- 歸檔和維護
- 格式化和版本控制

**應用場景**

此模式特別適用於需要知識累積和複合增長的領域：
- 個人知識管理和讀書筆記
- 深度研究和文獻綜述
- 企業內部 wiki 和知識庫
- 競爭分析和市場研究
- 學習路徑規劃

**技術優勢**

- **維護成本低**：LLM 不會厭倦更新鏈接和交叉引用
- **知識複合增長**：新資訊有機地與既有知識整合
- **版本控制內建**：基於 git 的 Markdown 天然支持版本歷史
- **可查詢性強**：結構化的頁面比原始文檔更易搜索
- **一致性高**：單一真理來源，避免知識分散和矛盾

## 與傳統方法的對比

| 特性 | RAG | LLM Wiki |
|------|-----|----------|
| 知識編譯 | 每次查詢 | 一次，持續維護 |
| 一致性 | 低（可能矛盾） | 高（單一來源） |
| 維護成本 | 低 | 中等（但高度可自動化） |
| 查詢延遲 | 低 | 中等 |
| 知識深度 | 淺層（片段式） | 深層（綜合式） |
| 可擴展性 | 好 | 優秀 |

## 實施建議

1. 開始時投入時間策劃 schema（結構）
2. 建立初始的高質量來源集合
3. 定期執行 lint 操作以維護品質
4. 逐步演進 wiki，而非追求完美的初始版本
5. 利用 git 的 branching 功能進行實驗性內容

## 未來展望

該模式代表了從被動檢索向主動知識構建的轉變，預示著 AI 在個人和企業知識管理中的新角色。
