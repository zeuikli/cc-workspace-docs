# ハーネスエンジニアリングは枠組みから始めよう

## はじめに

本記事は、CADDiで Quote というアプリケーション開発に携わるエンジニアが、ハーネスエンジニアリングの実装を通じた学習を記しています。

## この記事のゴール

著者の目標は「完璧なハーネスを最初から作り込まなきゃ」という心理的障壁を取り除くことです。「"とりあえず枠組みを作って、チームで育てていく"という発想」を共有し、気軽にハーネス整備を開始できる状態を作ることを重視しています。

## ハーネス整備のハードル

開発では、コーディングエージェントの出力品質が期待を下回る場面が多かったそうです。毎回「コードから自明なコメントは不要」といった指摘をしていましたが、実装ガイドラインが十分に整備されていませんでした。

複数の心理的障壁がありました：

- チーム間での認識統一の必要性
- ルール整備形式についての事前議論
- ルールファイルと実装PRの分離検討

最終的に Rules ファイルの整備に着手し、バックエンドの実装ルールを `.claude/rules/` ディレクトリに配置しました。

## Claude Code の Rules について

Claude Code には `CLAUDE.md` に指示を書く仕組みがありますが、量が増えると管理が困難になります。Rules システムでは `.claude/rules/` ディレクトリの Markdown ファイルでトピック別に分割管理できます。

YAML frontmatter の `paths` で特定ファイルパスを指定すると、条件付きルール（conditional rules）として機能します。例えば API 関連ファイルを扱う際のみ特定ルールが読み込まれます。

ルールは汎用ルール（レイヤー横断）とレイヤー別実装ルールに大別されます。

## ルール整備によって変わったこと

ルール整備後、実装中に「これもルールに盛り込みたい」という点が出現しました。そこでルール更新用スキル `/update-coding-rule` を作成しました。

このスキルにより、実装中の細かい指摘をコマンド一つで品質基準に反映できるようになりました。PR レビューコメント URL を渡すことで、該当スレッドを取得してルール化することも可能です。

## なぜうまくいったのか：枠組みの力

重要な気づきとして、ルール整備が成功した理由は「充実したルールファイル」ではなく、「どこに・どんな形で置くか」という枠組みの存在でした。

形式が決まっていると、更新ハードルが大幅に低下します。「これはここに追加すればいい」という道筋が見えることが、内容の充実度よりも重要でした。

## 始め方：枠組みだけ先に作る

著者の経験では、1日かけてルール全体を整備しましたが、それは不要でした。ほぼ空のルールファイル群だけで十分です。

提供されるプロンプト例では、プロジェクトアーキテクチャを分析し、Rules ファイルの骨組みを Claude Code に生成させています。

ルールドキュメントは2種類に大別されます：

1. **汎用ルール**：レイヤー横断の共通方針（例：coding-style.md、testing.md）
2. **レイヤー別ルール**：各レイヤーのサブディレクトリに実装・テストルール分離（例：usecase/usecase.md）

各ドキュメントには概要セクションを設け、目的を明示します。

## インデックスファイルの構成

`.claude/rules/` は"インデックス"として扱い、ルール本体への参照のみを記載します。frontmatter の `paths` で対象ファイルを指定し、本文でルール本体へのパスを記載します。

この構成により、conditional rules 読み込み時のトークン数を最小化しながら、必要に応じて詳細ルールにアクセスできます。

## Rules 以外にも枠組みは作れる

同じ発想は他の要素にも応用できます：

- **Skills**：最小限の機能で API stub 生成スキルを用意し、使用中にフィードバックを集める
- **Hooks**：各タイミング用の専用スクリプトを配置し、段階的に充実させる

「枠組みと責務が明確だと、更新・拡張のハードルが下がる」という原則は共通しています。

## 終わりに

ハーネスエンジニアリングは完璧な設計から始まるものではありません。内容がスカスカでも、「ここに足していける」という場所があれば、チーム内での気づきを反映できるようになります。

開発とハーネス整備を区別せず、同じタスクで完結させることで、ハーネスは自然に成長していきます。

## 繁體中文全文摘要

**原文語言**：日文（日本 CADDi 工程師分享 Harness Engineering 實作經驗）

### 核心命題：從框架開始，不求完美

最大的心理障礙是「覺得要完美設計才能開始」。正確的心態是：**先建立框架，讓團隊共同培育**——輕鬆開始 harness 整備比一次建好更重要。

### 常見的心理障礙

- 需要統一團隊認知
- 需要事先討論規則整備形式
- 需要考慮規則檔與實作 PR 的分離

最終行動：著手整備 Rules 檔案，將後端實作規則放置在 `.claude/rules/` 目錄。

### Claude Code Rules 系統

- `CLAUDE.md` 隨內容增加而難以管理 → Rules 系統允許在 `.claude/rules/` 按主題分割管理
- YAML frontmatter 的 `paths` 指定特定檔案路徑 → **Conditional Rules**（條件式規則）：例如只在處理 API 相關檔案時載入特定規則
- 兩大類規則：**汎用規則**（跨層通用方針）+ **層別實作規則**（各層的實作/測試規則）

### `/update-coding-rule` Skill：即時反饋閉環

整備規則後，實作中會出現「這也想加進規則」的情況。作者建立了規則更新用 skill：
- 一個指令就能將實作中的細微指摘反映到品質基準
- 傳入 PR review 評論 URL → 取得該討論串 → 直接規則化

### 框架的力量：「放在哪裡」比「放什麼」更重要

成功的關鍵洞見：規則整備成功的原因不是「豐富的規則檔案」，而是「**決定放在哪裡、用什麼形式放**」這個框架的存在。

形式一旦確定：
- 更新門檻大幅下降
- 「這個加在這裡就好」的路徑清晰
- 內容的充實程度比框架的存在更次要

### 實際建立方式

**從幾乎空的規則檔案開始就足夠**——用 prompt 讓 Claude Code 分析專案架構，生成 Rules 檔案的骨架。

規則文件分類：
1. **汎用規則**：coding-style.md、testing.md（跨層共通方針）
2. **層別規則**：各層的子目錄分別放實作規則 + 測試規則

Index 檔案設計：`.claude/rules/` 作為「索引」，只記載到規則本體的參照，最小化 conditional rules 載入時的 token 數。

### 同樣原則適用 Skills 和 Hooks

- **Skills**：先準備最小功能的 API stub 生成 skill，在使用中蒐集反饋
- **Hooks**：各時機的專用 script 先配置好，再逐步充實

「框架和職責清楚 → 更新/擴充門檻下降」是共通原則。

### 核心哲學

開發與 harness 整備不分離——在**同一個任務中完結**，harness 就會自然成長。完美設計不是開始的前提，「這裡可以加東西」的確定感才是。
