# MemGPT: Towards LLMs as Operating Systems

- **arXiv**: https://arxiv.org/abs/2310.08560
- **發表**: 2023（更新至 2024-02）
- **作者**: Charles Packer et al.
- **收錄日期**: 2026-05-01
- **重要性**: Priority A — Long-running agent 記憶管理的基礎架構；虛擬 context 管理的開創性論文

---

## 核心類比：LLM 作為操作系統

傳統 OS 管理有限的 RAM（快速但有限）+ 無限的磁盤（慢但無限）。

MemGPT 將同樣的原理應用到 LLM：

| OS 概念 | MemGPT 對應 |
|---------|-----------|
| RAM（快速、有限） | Context Window（main context） |
| 磁碟（慢、無限） | External Memory（recall + archive） |
| 虛擬記憶體 | Virtual Context Management |
| 頁面置換 | 動態 context 移入/移出 |

---

## 三層記憶架構

### Layer 1：Main Memory（主記憶體）
- 存放內容：當前對話、working notes、最近事件
- 對應：LLM 的 context window
- 限制：有限 token 預算

### Layer 2：Recall Storage（回憶資料庫）
- 存放內容：完整的過去對話歷史
- 存取方式：語義搜索（embedding）+ 時間戳
- 讀/寫：agent 可以主動查詢

### Layer 3：Archival Storage（檔案存儲）
- 存放內容：長期知識、文件、筆記
- 存取方式：向量索引（FAISS 等）
- 讀/寫：結構化知識庫

---

## 控制流設計：Interrupts 機制

```
LLM 決定動作 → 發送 interrupt 訊號 →
  - LOAD_FROM_RECALL: 從回憶 DB 載入特定記憶
  - SEARCH_ARCHIVAL: 搜索長期存儲
  - PUSH_TO_ARCHIVAL: 存入長期記憶
  - PAUSE_HEARTBEAT: 告知系統需要更多時間
```

這讓 LLM 自己管理記憶的讀寫，而不是靠外部系統強制執行。

---

## 核心應用場景

1. **分析超長文件**：比 context window 大 10x+ 的文件，按需載入相關片段
2. **長期對話 agent**：記住用戶的偏好、歷史決策、長期目標
3. **多 session 任務**：agent 能「記得」上個 session 發生了什麼

## 對 Harness Engineering 的影響

### 洞見 #1：Context 管理是 harness 的核心責任，不是 LLM 的責任
MemGPT 明確把「決定什麼留在 context、什麼存到外部」作為系統責任，而不是讓 LLM 自己決定。

### 洞見 #2：讀寫記憶的 API 設計至關重要
Interrupts 的設計方式（LLM 主動發出信號而非被動接收）讓 agent 對記憶有更多控制權。這啟發了後來 harness 設計中的「agent-initiated context management」。

### 洞見 #3：三層記憶不是可選優化，是長期 agent 的必要架構
任何需要跨 session 工作的 agent 都需要類似的分層記憶架構。
